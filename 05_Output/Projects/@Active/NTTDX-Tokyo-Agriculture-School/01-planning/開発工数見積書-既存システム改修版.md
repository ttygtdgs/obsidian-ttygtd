#### 各項目の説明

| 項目 | 説明 |
|------|------|
| **既存システム分析・理解** | 引き継いだソースコードの構造把握、データベース設計の理解、既存機能の仕様確認、開発・テスト環境のセットアップを行います。既存システムを安全に改修するための基盤作りです。 |
| **改修設計・仕様調整** | 改修が必要な箇所の特定、改修による影響範囲の分析、改修設計書の作成、ステークホルダーとの仕様合意を行います。手戻りを防ぐための重要な工程です。 |
| **フロントエンド改修** | ユーザーが操作する画面（申請フォーム、管理画面等）のUI/UX改善、機能追加・修正、スマートフォン対応（レスポンシブ化）を行います。 |
| **バックエンド改修** | サーバー側の処理（API、申請処理ロジック、承認フロー等）の改修、データベーステーブルの変更やデータ移行作業を行います。 |
| **テスト・検証** | 改修した各機能の単体テスト、機能間連携の結合テスト、ユーザー受入テスト（UAT）の支援と不具合修正を行います。品質担保のための重要な工程です。 |
| **ドキュメント整備** | 改修内容の説明書作成、運用マニュアルの更新、システム仕様書の更新を行います。運用引継ぎに必要な資料を整備します。 |
| **環境構築・デプロイ** | 本番環境のセットアップ、改修済みシステムのデプロイ（リリース）作業、動作確認を行います。 |

## 前提条件・制約事項

### 前提条件
- 既存システムのソースコード一式が提供されること
- 既存システムの仕様書・設計書（あれば）が提供されること
- 既存システムのテスト環境・本番環境へのアクセス権が提供されること
- 既存システムの技術担当者からの引継ぎ対応が可能なこと

### 技術スタック（想定）
- 既存システムの技術スタックに準拠
- 引継ぎ後に詳細確定

### 除外事項
- AI機能の新規追加（今回は不使用、別途オプション）
- iOS/Androidアプリ開発
- 大規模な機能追加・リニューアル
- 既存システムの完全リプレイス

### リスク要因
- **技術的負債**: 既存システムのコード品質によっては改修工数増加の可能性
- **仕様変更**: 要件変更時は別途見積もり
- **ドキュメント不足**: 既存仕様書が不十分な場合、分析工数増加の可能性

### 推奨シナリオ

**既存システム改修版がおすすめのケース:**
- 予算を抑えたい場合
- 既存システムの品質が一定以上の場合
- 短期間でリリースしたい場合
- 大幅な機能追加が不要な場合

**スクラッチ開発版がおすすめのケース:**
- 既存システムの技術的負債が大きい場合
- 大幅な機能追加・改善が必要な場合
- 長期的な保守性・拡張性を重視する場合
- モダンな技術スタックを採用したい場合
