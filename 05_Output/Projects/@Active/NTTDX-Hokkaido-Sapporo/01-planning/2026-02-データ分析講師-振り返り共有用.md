# データ分析講師としての気づき — 札幌アイデアソン振り返り

**対象**: 2026年2月3日 北海道科学大学 データ活用アイデアソン
**役割**: データ分析講義パート担当（講義＋Looker Studioハンズオン）
**目的**: 次年度実施に向けた良かった点・改善点の整理

---

## 良かった点

### 1. 全学生がデータに触れ、発表まで完走できた
- 全チームが「データを見る → 特徴を捉える → アイデアに結びつける → 発表する」という一連のプロセスを達成
- データを起点に思考する体験そのものを提供できた

### 2. 学生のリアルなレベル感を把握できた
- 想定よりもITリテラシーが高いことが判明。Looker Studioのフィルター操作等はほぼ説明不要なレベルだった
- この発見は次年度の講義設計における最も重要なインプット

### 3. 学年・学部を超えた交流が自然に生まれた
- 異なるバックグラウンドの学生同士がチーム内で自然にコミュニケーションを取り、協働できていた

### 4. 初開催の2日間プログラムとして成立した
- ゼロからの企画で全体を構築・実施し完遂。満足度が非常に高く、学生・ステークホルダー双方から好評

### 5. データの特徴を見てアイデアを考える体験が成立した
- 完全なデータドリブンとまではいかないが、データの特徴を見た上でサービスアイデアを考えるという流れは機能した

---

## 改善点

### 1. 学生のレベルに対して講義の難易度が低すぎた
- **事象**: 学生のレベルが想定より高く、基本的なツール操作の説明が不要な場面が多かった。結果として時間を持て余すケースが発生
- **原因**: 初開催で学生のレベルが予測できず、難易度設計が保守的になった
- **示唆**: 次年度は事前のレベル把握と、当日のリアルタイム調整の仕組みが必要

### 2. Looker Studioハンズオンが「ツール操作」に終始した
- **事象**: フィルター操作などの基本的なUI操作の説明が中心で、学生にとっては既知の内容が多かった
- **本来やるべきだったこと**: ツールの使い方ではなく「データの投入方法」「データの見方・読み方」に時間を割くべきだった。同じデータを複数の切り口（時系列・比較・分布等）で見るワークがあれば、データの特徴をより深く捉えられた可能性がある

### 3. 「データをどう使えばいいか」のヒントが不足していた
- **事象**: 今回、学生には「スポーツに興味がない人にどうアプローチするか」といった課題を設定した上で、選手の成績や試合データを渡した。しかし、たとえば「ピッチャーの球種別の被打率」のようなデータを見ても、それをどうすれば「興味がない人を呼ぶ」ことにつなげられるのか、その橋渡しの考え方が示されていなかった
- **具体例**: 同じ「選手が1試合で10km走る」というデータでも、使い方は複数ある
  - 「札幌駅から大通公園を4.5往復」と言い換えて、スポーツを知らない人にも伝わる話題にする
  - 試合日の観客の動きデータと組み合わせて、周辺の飲食店にとってのビジネスチャンスを見つける
  - 「だからこの選手はすごい」というストーリーの裏付けとして使う
- **結果**: こうしたデータの使い方の引き出しが提示されなかったため、チームによってデータ活用の深さにばらつきが出た
- **示唆**: 次年度は「このデータはこういう使い方もできる」という具体例を講義の中でいくつか見せておくことで、学生がデータと課題をつなげるヒントを得やすくなる。以下の5つの「データの使い方の引き出し」を整理した。

#### データの使い方の引き出し（5パターン）

以下、今回のアイデアソンで実際に使えたデータを例に示す。

---

**引き出し1: 身近なものに言い換える**

データの数値をそのまま見せるのではなく、誰でもイメージできるスケールに変換する。スポーツに詳しくない人にも「すごさ」や「面白さ」が伝わるコンテンツになる。

| 元のデータ | 言い換えた例 |
|-----------|------------|
| Jリーグ：選手のトラッキングデータで「1試合の走行距離が10km」 | 「札幌駅から大通公園を4.5往復している」 |
| 野球：投手の球速が150km/h | 「マウンドからホームベースまで0.44秒。まばたき1回分で届く」 |
| バスケ：ダンクの成功数とリバウンド数 | 「1試合でビル3階分の高さを飛んでいる」 |
| Jリーグ：トラッキングデータの速度(Speed) | 「トップスピードの瞬間、札幌市内の制限速度を超えている」 |
| 野球：投手の投球数が1試合120球 | 「腕を振る回数はテニスの1セット分に相当」 |

---

**引き出し2: 別の目的で読み替える**

スポーツのためのデータを、別の視点から見直す。同じデータでも「何のために見るか」を変えるだけで、新しい発見がある。

| 元のデータ | スポーツとしての見方 | 別の目的で見ると |
|-----------|-------------------|----------------|
| 野球・バスケ：試合ごとの観客動員数（attendance） | チームの人気度を測る | 試合がある日に地域にどれだけ人が集まるかを示す「地域の人流データ」として使える |
| Jリーグ：試合情報の天候データ（晴/曇/雨/雪） | 試合のコンディション分析 | 天候ごとの来場者数を見れば「雨の日にどんなサービスがあれば来てもらえるか」のヒントに |
| 野球：選手プロフィールの出身地（born_local）・出身高校 | 選手のバックグラウンド | 「北海道出身の選手」で絞り込めば、地元ファンへのアピール材料になる |
| バスケ：試合の開始時刻・終了時刻 | スケジュール管理 | 「試合が終わる時刻」がわかれば、周辺飲食店の準備タイミングの目安になる |

---

**引き出し3: 2つ以上のデータを組み合わせる**

1つのデータだけでは見えないことが、別のデータと重ねると見えてくる。

| データA | データB | 組み合わせると見えること |
|--------|--------|---------------------|
| 野球：試合日・観客動員数 | Jリーグ：試合情報の天候データ | 「雨の日は来場者が減る。では雨の日限定の施策は？」という問いが立つ |
| 野球：選手の出身地（born_local） | itown：地元企業の業種・住所 | 「北海道出身の選手×地元の飲食店」でコラボ企画の候補が見つかる |
| バスケ：試合ごとの観客動員数 | バスケ：試合の曜日・開始時刻 | 「金曜夜の試合は動員が多い」→ 周辺店舗の営業時間延長の根拠に |
| Jリーグ：ゴール・アシストのイベントデータ | Jリーグ：トラッキングの速度データ | 「ゴール前の10秒間に選手の速度がどう変わるか」→ 盛り上がりの瞬間を可視化 |
| itown：スタジアム周辺の飲食店数 | バスケ：試合終了時刻 | 「試合終了後に開いている飲食店が半径500mに何件あるか」→ アフターゲーム体験の設計 |

---

**引き出し4: アイデアの裏付けに使う**

先にアイデアを考えてから、「このアイデアが正しい理由」をデータで説明する。データを「分析する」のではなく「説得材料にする」という使い方。発表の説得力が大きく上がる。

| アイデア | データによる裏付け |
|---------|-----------------|
| 「家族向けの観戦イベントをやりたい」 | バスケの観客動員数データで「土日の午後の試合は動員が多い」→ 家族層が来やすい時間帯である証拠 |
| 「試合前にスタジアム周辺を楽しめる仕掛けがほしい」 | itown：スタジアム半径1km以内に飲食店が○件ある → 連携先候補がこれだけあるという根拠 |
| 「地元出身選手を前面に出したプロモーションをしたい」 | 野球：選手プロフィールの出身高校・出身地で「北海道出身が○人」→ 地元密着のストーリーが作れる |
| 「得点シーンをSNSで拡散したい」 | Jリーグ：イベントデータのゴール・アシスト数で「1試合平均○回の盛り上がりポイント」→ コンテンツ量の見積り |
| 「雨の日でも来たくなる仕組みがほしい」 | Jリーグ：天候データ×観客数で「雨の日は○%減る」→ 雨の日限定特典の必要性を数字で示す |

---

**引き出し5: 他の分野のヒントにする（上級向け）**

スポーツデータのパターンを、まったく別の分野の課題解決に当てはめる。難易度が高いため、余裕のあるチーム向けのチャレンジ課題として位置づける。

| スポーツデータのパターン | 別の分野への応用 |
|----------------------|----------------|
| Jリーグ：パス成功率やパス方向のデータ → チーム内の連携パターンが見える | 会社や組織の中で「誰から誰に情報が流れているか」を考えるヒントになる |
| バスケ：シュート座標（X座標・Y座標）→ コート上のどこからシュートが多いかがわかる | 店舗の売り場で「どの棚の前にお客さんが集まるか」の分析と同じ発想 |
| 野球：打球方向（pull/center/opposite）と打球の強さ(soft/med/hard) → 打者の傾向が見える | 顧客の行動パターン（いつ・どこで・何を買うか）を分類する考え方と似ている |

### 4. 札幌市に「データが活用された」と伝わる設計が不足していた
- **良かった点**: 発表フォーマットに「どんなデータで」という項目を入れたことで、学生がデータを使った形で発表する構造にはなっていた。この仕組み自体は次年度も継続すべき
- **事象①（意味づけ・示唆だしの解説不足）**: データを可視化するところまではできていたが、その先の「このデータからどういう意味が読み取れるか（意味づけ）」「だからどうすべきか（示唆だし）」のステップについて、講義でもう少し丁寧に解説すべきだった。データを見せることと、データから何が言えるかを導き出すことは別のスキルであり、その橋渡しのガイドが不足していた
  - 例：「観客動員数が雨の日に30%減る」というデータを見せるだけでなく、「つまり雨の日は来場のハードルが高い → 雨の日に来る理由を作れば伸びしろがある」という意味づけ・示唆の出し方を見せる
- **事象②（ペルソナ設計への巻き込み不足）**: ペルソナ（誰の課題を解くか）は運営側で事前に設定しており、札幌市はその過程に関与していなかった。もし札幌市自身がペルソナ設計に参加していれば、「自分たちが設定した課題に対して、学生がデータを使ってこういうアプローチをした」という見方ができ、データ活用の実感を得やすかった可能性がある
- **示唆**: 次年度は2つの面から改善が考えられる
  - **講義での「意味づけ→示唆だし」の解説強化**: データを可視化した後に「これは何を意味するか」「だからどうすべきか」を考えるステップを、具体例を使って講義の中で実演する。学生が発表で「データ → 意味づけ → 施策」の流れを自然に語れるようになれば、札幌市にもデータ活用のプロセスが伝わりやすくなる
  - **ペルソナ設計への札幌市の参加**: 事前ワークショップで札幌市担当者と一緒にペルソナを設計することで、「自分たちの課題がデータで解かれていく」過程を体感できる仕組みにする

## 次年度への提案

### 講義設計の改善

| 項目 | 今回 | 次年度提案 |
|-----|------|----------|
| ハンズオンの焦点 | ツール操作（フィルター等） | データの見方・投入方法 |
| 難易度調整 | 固定（保守的） | 冒頭アセスメント＋段階的開放 |
| データ活用ガイド | なし | 5パターンの活用型を具体例付きで提示 |
| 上級者対応 | なし | ストレッチゴール（追加分析・プロトタイプ） |

### 冒頭アセスメント＋段階的開放の導入
- プログラム冒頭5分で簡単な操作チェック（例：「この平均値を出してください」）を行い、リアルタイムにレベルを測定
- 講義スライドにスキップポイントを事前マークし、レベルが高ければ基本説明を飛ばして発展的な内容へ
- ハンズオンを段階構造（操作 → 可視化 → 分析 → 投入）にし、早い人は先に進める

### 札幌市の巻き込み設計
- 事前にペルソナ設計ワークショップを札幌市担当者と実施し、課題設定の当事者にする
- イベント中は審査員・コメンテーターとして参画し、学生への直接フィードバックの機会を設ける
- イベント後に学生アイデアの報告会を開催し、PoC検討の場を設ける
