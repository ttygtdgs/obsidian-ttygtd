# AI系開発PJにおける問題解決のためのサーベイ

> Main Author: Yuto Mori（元松尾研究所PM）
> 別途、動画による実践編あり

---

## AI系開発PJの全体の流れ

### Step 1. ドメイン知識の把握

- 専門用語を知らなければ検索すらできない
- **目標**: その分野で3年働いた人レベルの「わかってる感」を出せること
- 初期はトンチンカンなことを言って当然 → 仮説検証を高速で回してソリッドな解へ

### Step 2. 超具体的な解法パターンを作り込む

- NG: 「機械の異常検知できれば良いですよね〜」（曖昧）
- OK: 「aという機械のbという部品の異常検知をc秒で検知し、○○粒度のレポートで人間に伝えれば、d%早く異常発見でき、e%事前特定可能。リスク低減と生産ラインストップ回避で1000万円/年の改善見込み」
- **最低2-3個の具体的アプローチを作る** — ディテールがないと議論は進まない
- 技術サーベイと表裏一体

### Step 3. 「本当の課題は何か？」を対話で明らかにする

- Step 1-2と**並行して**実施
- クライアントが課題と思っていることが真の課題でないことは多い
  - 例: 患者が「お腹が痛い」→ 実は心臓に病気（医者＝我々、患者＝クライアント）
- 企業ごとに課題整理力は大きく異なる。先方の画に乗ってうまく行かないことの方が多い
- **サイクル**: サーベイ → スライドにまとめ → 客に提案 → フィードバック → 再調査
- 1度のサイクルでの成功確率は5%未満 → 繰り返し＆チームで確率を上げる

### 注意: 特許・ライセンス

- 競合特許の調査は重要だが、訴訟時に不利な証拠にならないよう注意
- 詳細は法務専門家に相談

## 技術的解法を論文から調べる方法

### 1. Google Scholarで検索

- **ドメイン専門用語 + やりたいこと** で検索
  - 例: `retail + demand + time-series forecasting`
- 「ググラビリティが高い」= vivid な単語を見つけることが鍵
- ドメイン特化の学会・雑誌を見つけたらメモ
- 日本語 Qiita はとっかかりにはOKだが、一次情報ではないので注意
  - 原論文に書いていないことを言っている可能性あり
  - クライアントへは一次情報から伝えること

### 2. 学会の採択論文から探す

- `学会略称 + 年 + accepted papers` で検索
  - 例: `NeurIPS 2020 accepted papers`
- ページ内でキーワード検索
- 普段からどの学会が何に強いか頭に入れておく
- 別分野の知人にtop tier学会を聞いておくと良い

### 3. 論文を素早く読んでまとめる

- **初期**: 1本4時間かかっても仕方ない。特にIntroductionを丁寧に読む
- **30本ほど読んだ後**: 脳内マップ形成 → Abstract + Experiments + Conclusionを中心に高速読み
- **まとめ方**: 落合陽一スタイルが有名
  1. どんなもの？
  2. 先行研究と比べてどこがすごい？
  3. 技術や手法のキモはどこ？
  4. どうやって有効だと検証した？
  5. 議論はある？
  6. 次に読むべき論文は？
- Notion/Scrapboxでテンプレート化して蓄積

### 4. 「土台となる論文」を見つける

- やりたいタスクに近い論文は大抵存在する
- チェックポイント:
  - 実験設定・データ量・到達精度
  - 評価指標の妥当性
  - 著者のバックグラウンド（どの企業が注目しているか）
- **サーベイ論文**は体系的だが、今解きたい問題に対して網羅的とは限らない

### 5. 参考文献を帰納的に読む

- 微妙な論文でも参考文献には王道論文が引用されていることが多い
- ただし、参考文献だけでは解決策は見つからないことも多い
  - 先人が解決済みなら既に解かれている
  - **ビジネス・ドメイン知識で優位性を出してもよい**（研究と異なる点）
- **他分野からの知見が役立つことがある** → 抽象化する能力が重要
  - 例:「この時系列データ、NLP的に扱えるのでは？」

## Kaggle過去コンペからの調査

- コンペ終了後にdiscussionに解法が投稿される
- workshopで短い論文が出ることもある（例: NeurIPSワークショップ）
- 普段からキャッチアップして「あのコンペっぽいな」と想起できるのが理想
- **コンペ参加で得られる肌感**:
  - データ前処理の難しさ
  - 評価指標の良し悪し
  - リークの発生パターン
  - ドメイン知識に基づく特徴量
  - 基本モデルの選定
  - タスクの難易度感
- 検索は Competitions に限定して notebook/discussion を読みに行く

## 競合調査

### なぜ競合が存在するか

- AI投資の動機は大きく2つ:
  - **業界1番手**: 他社追随を許さない地位確立
  - **業界2番手以降**: 他社のAI導入に焦り
- 2022年時点で1番手のAI投資は一巡 → 案件は業界トップ企業とは限らない

### 競合の把握方法

- クライアントのビジネスモデル理解が前提
- 「競合はいない」→ 自分の理解不足を疑うべき
- クライアントとの会話中に自然と出てくることが多い
- プロダクトベースで考える: 同じペインを解決する別手段 = 競合

### 競合の技術を調べる

- プレスリリース・ブログ・ニュースから技術を逆探知
- プロダクト紹介ページから使用技術を推測
- 隠されている部分・誇大な部分のカラクリを考える
- **差別化ポイント**を常に意識（自社の強み/弱みの軸）

## 教科書の重要性

- 「多分あの分野だろうな」と思える力が重要
- 大学のシラバス・カリキュラムを眺めると分野の全体像が掴める
- 実務で使いうる隣接分野の例: 制御論 / 離散最適化 / 因果推論 / 心理統計 / レコメンド
- 機械学習プロフェッショナルシリーズの該当巻は最低限頭に入れる
- **「知は力」** — 教科書で基盤を固めてから最新論文にキャッチアップできれば最強
