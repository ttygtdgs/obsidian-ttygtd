# 自治体案件 提案方針整理 — 壁打ちメモ

> **目的**: NTTDXPNとして自治体の生成AI案件に対する提案方針を整理する
> **形式**: 壁打ちで出た論点・フレームワークの記録
> **日付**: 2026-02-25

---

## 1. 核心の問い

**自治体案件でNTTDXPNは「何で稼ぐのか」が定まっていない。**

- OpenAI/Gemini等のLLM APIを使って何かを構築する場合と、それらの上に構築されたSaaSが求められる場合がある
- NTTDXPNはAPI活用の技術力はあるが、自前のSaaS商材を持っていない
- 自治体案件ではSaaS+SI+研修がセット発注されることが多く、SaaS商材がないと応札すらできないケースがある

---

## 2. バリューチェーン上の位置

```
LLMベンダー → プラットフォーム/SaaS → SI/導入支援 → 自治体
(OpenAI等)    (ミンクスプラス等)        (ここがDXPN?)    (エンド)
```

### 自治体が買う3つのもの

| 層 | 顧客が買うもの | 具体例 | 誰が提供するか |
|---|---|---|---|
| (1) | **プロダクト** — AI環境そのもの | SaaSライセンス、チャット基盤 | SaaSベンダー |
| (2) | **フィッティング** — うちの組織に合わせる | RAG構築、データ整備、業務設計 | **SI** |
| (3) | **定着** — 使えるようにする | 研修、伴走支援、効果検証 | **コンサル/SI** |

**DXPNは(2)(3)はできるが、(1)を持っていない** — これが構造的な課題。

### 3,000万円規模案件（長野県型）の収益構造イメージ

| 項目 | 金額 | DXPNの利益 |
|---|---|---|
| SaaSライセンス | 1,200-1,500万 | 薄利（ベンダーの売上） |
| RAG構築 | 500-800万 | ◎ |
| 研修・定着支援 | 200-300万 | ◎ |
| 運用保守 | 300-500万 | ◎（ストック型） |
| PM | 200-300万 | ○ |

## 3. ニーズの分類: 汎用 vs 特化

### 汎用ニーズ
- 「全庁にAIチャット環境を」「職員がAIを使えるようにしたい」
- 既製品SaaSで解決できる
- 競争軸: 製品力・価格・認証・実績 → **プロダクト勝負**

### 特化ニーズ
- 「この業務の、この課題をAIで解きたい」
- 業務特化RAG、ワークフロー自動化、既存システム連携、住民向けAI
- 既製品では対応できない → 作る必要がある
- 競争軸: 技術力・業務理解・提案力 → **ソリューション勝負**

### DXPNの立ち位置
- **汎用**: 商材を持っていない → 担がないと戦えない
- **特化**: API活用構築力がある → 自分たちが主役

### 市場の時間軸

| フェーズ | ニーズ | 主流 | DXPNの強み |
|---|---|---|---|
| 第1波（今） | 汎用 — まず全庁AI環境整備 | SaaS導入 | 活きにくい |
| 第2波（これから） | 特化 — 環境の上で業務特化AI | カスタム構築 | **活きる** |
| 第3波（将来） | AI前提の業務改革 | 伴走型コンサル | 活きる |

> 今は市場と自分たちの強みがズレているが、市場が成熟すれば合ってくる可能性がある。

### 戦略オプション
- (a) 第1波にも乗る → SaaSを担いで、かつ特化を付加価値にする
- (b) 第2波に集中する → 今は汎用をスキップ、特化案件だけ取る
- (c) 第1波を入口に第2波につなげる → 研修/PoCで入って特化構築にアップセル

## 4. 提供パターン A〜D

「DXPNが何を届けるか」で4パターンに分類。

| # | パターン | 何を届けるか | 使うもの | 効果 |
|---|---|---|---|---|
| **A** | **AI環境** | 全員がAIを使える基盤 | SaaS or Dify | 利用可能にする |
| **B** | **AI機能** | 特定業務向けのAI機能 | LLM API + 開発 | 業務を支援する |
| **C** | **AI自動化・高度化** | 業務プロセスにAIを組み込む | n8n + LLM API + UiPath | 効率化+質の向上 |
| **D** | **AIコンサル** | 知識・戦略・定着支援 | 人（研修、伴走） | 人を変える |

### 汎用/特化 × 提供パターンの掛け合わせ

| | 汎用ニーズ | 特化ニーズ |
|---|---|---|
| **A: AI環境** | 全庁チャットAI | 部署別AI環境 |
| **B: AI機能** | 庁内文書RAG | 福祉相談RAG、議事録要約 |
| **C: AI自動化・高度化** | --- | **帳票処理、データ入力、報告書自動生成** |
| **D: AIコンサル** | AI研修、ロードマップ | 業務特化ワークショップ |

> **Cはほぼ全て「特化」** — 業務フローは組織ごとに違うため。

---

## 5. BとCの違い

核心の違い = **「誰がAIを動かすか」**

### B: AI機能 — 人がAIを使う

```
人 → [AI] → 人
      ↑
  「〇〇を教えて」「要約して」
```

- 人が能動的にAIに向き合う（インタラクティブ）
- 使うかどうかは人次第
- 技術: LLM API + UI（Dify等）

### C: AI自動化・高度化 — プロセスの中でAIが動く

```
トリガー → [n8n] → [AI] → [UiPath/外部システム] → 結果
            ↑
  業務イベント（書類到着、データ更新、定時バッチ等）
```

- AIがプロセスの一部として組み込まれている
- 人が意識しなくてもAIが動く
- 技術: n8n + LLM API + UiPath + 外部連携

### 具体例で比較

| ユースケース | B: AI機能 | C: AI自動化・高度化 |
|---|---|---|
| 議事録 | 「この議事録を要約して」と人が指示 | 会議終了後、自動で要約→配布 |
| 文書検索 | 「〇〇の規定は？」と人がRAGに聞く | 申請到着時、関連規定を自動引用→判断支援 |
| データ分析 | 「このCSVを分析して」と人が投げる | 月次データ更新時、自動分析→レポート生成 |
| 品質チェック | 「この文章をレビューして」と人が依頼 | 起案文書提出時、AIが自動チェック→修正提案 |

### 自動化と高度化

- **自動化**: 人がやっていた処理をAIが代替（帳票→データ抽出→転記）
- **高度化**: AIが判断・分析を入れてプロセスの質を上げる（報告書自動チェック、予測分析→示唆提示）

> CにAIが埋め込まれた方が効果が持続する — 人が使わなくてもAIが動くから。

## 6. DXPNの強み/弱みマッピング

```
A: AI環境     → △  SaaS商材がない（担がないと戦えない）
B: AI機能     → ✅  API活用で開発できる
C: AI自動化   → ✅✅ n8n+UiPath+APIで差別化できる ← ここが武器？
D: AIコンサル → ✅  人が対応
```

**強み: B, C, D（特化寄り）**
**弱み: A（SaaS商材がない）**

> **C: AI自動化**がDXPN固有の強みになる可能性。SaaSベンダー（exaBase/GaiXer等）はAI環境は提供するが、n8n+UiPathでの業務フロー自動化まではカバーしない。

### SaaSを担ぐ場合の比較

| | SaaSを担いだ場合 | 担がない場合 |
|---|---|---|
| 応札できる案件数 | 多い | 少ない（B型のみ） |
| 案件単価 | 大きい（SaaS+SI込み） | 中程度（SI部分のみ） |
| 利益率 | 低〜中（SaaS部分は薄利） | 高（全部自分の売上） |
| 顧客との関係 | 一次請けになれる | 一次請けになれる |
| 参入障壁 | パートナー契約が必要 | 技術力のみ |

## 7. 今後の議論論点

### 戦略判断
1. **「汎用」にも手を出すか** — 汎用SaaSを担ぐコスト（パートナー契約・薄利）に見合うか
2. **「特化」は今どれくらいあるか** — 自治体の特化案件のボリューム感
3. **汎用→特化のアップセル導線** — 第1波で入って第2波に繋げるモデルは成立するか

### 組織・関係性
4. **NTT東日本との棲み分け** — 汎用はNTT東、特化はDXPNという分業は可能か
5. **NTTDXPNのアイデンティティ** — 「AIのSI屋」か「AI技術のプロフェッショナル」か

### 外部環境
6. **源内（ガバメントAI）の影響** — 汎用が無料化されたら特化だけが残る？
7. **自治体 vs 民間** — 自治体は参入障壁（認証）があるが一度入ると固い。民間は自由だが競争激しい
