# 自治体案件の種類 × NTTDXPNの提案の種類

> **目的**: 自治体の生成AI関連案件を種類別に分類し、それぞれに対してNTTDXパートナーのAIチームがどのような提案形態で臨めるかを整理する。
> **背景**: NTTDXPNは自前の生成AI SaaSプロダクトを持たないため、案件種類ごとに「既存SaaSを担ぐ」「スクラッチ開発」「コンサル/研修」の使い分けが必要。
> **日付**: 2026-03-02

---

## 1. 前提: NTTDXPNの構造的制約

NTTDXPNのAIチームは以下の特性を持つ。この制約を前提に、案件種類ごとの戦い方を整理する。

| 項目 | 状況 |
|------|------|
| **自社SaaS** | なし（生成AIのチャット基盤SaaSを持っていない） |
| **技術力** | LLM API活用、RAG構築、Dify/n8n/UiPathでの業務自動化に強み |
| **提供可能範囲** | SI（設計・構築・運用）、コンサル、研修 |
| **NTTグループ連携** | NTT東日本の支店ネットワーク、tsuzumi等のグループ資産を活用可能 |

### バリューチェーン上の位置

```
LLMベンダー → SaaSベンダー → SI/導入支援   → 自治体
(OpenAI等)    (exaBase等)     (★ここがDXPN)    (エンド)
```

**自治体が購入する3つのレイヤー**:

| レイヤー | 内容 | DXPNの対応力 |
|----------|------|:---:|
| (1) プロダクト | AI環境そのもの（SaaSライセンス） | △ 担ぐ必要あり |
| (2) フィッティング | 組織に合わせた構築（RAG、業務設計、連携） | **◎** |
| (3) 定着 | 研修、伴走支援、効果検証 | **◎** |

## 2. 自治体案件の6類型

入札王データ（887件）および公募調査から、自治体の生成AI関連案件を以下の6類型に分類する。

### 類型一覧

| # | 案件類型 | 概要 | 年間件数（推計） | 単価帯 | SaaS必要度 |
|---|---------|------|:---:|------:|:---:|
| **I** | **全庁AI環境導入** | 全職員が使える生成AIチャット環境の構築 | 80〜100件 | 300〜3,000万円 | **高** |
| **II** | **AIチャットボット構築** | 住民向けFAQ・問い合わせ対応Bot | 40〜50件 | 200〜1,000万円 | **高** |
| **III** | **RAG・ナレッジ検索構築** | 庁内文書・条例・マニュアルの検索システム | 10〜20件 | 500〜2,000万円 | **中** |
| **IV** | **業務特化AI開発** | 特定業務向けAI（議事録要約、申請処理等） | 20〜30件 | 300〜2,000万円 | **低** |
| **V** | **AI研修・人材育成** | 職員向けAIリテラシー研修、ワークショップ | 40〜50件 | 30〜500万円 | **なし** |
| **VI** | **AI戦略・ロードマップ策定** | AI活用計画策定、PoC設計、交付金申請支援 | 10〜15件 | 200〜800万円 | **なし** |

---

### 類型 I: 全庁AI環境導入

**自治体が求めているもの**:
- 全職員がセキュアに生成AIを使えるチャット環境
- LGWAN対応またはα'モデル準拠のセキュリティ
- 利用ログの管理・監査機能
- 簡単なUI（プロンプト不要の「モード機能」等）
- ガイドライン整備・運用ルール策定を含む

**典型的な公募要件**:
- 「庁内で安全に利用できる生成AIツールの導入及び運用業務」
- SaaSライセンス＋導入支援＋研修がセット発注されることが多い
- 利用者数に応じた月額課金 or 年額一括

**主要な競合・既存プレイヤー**:

| プレイヤー | サービス名 | 特徴 |
|-----------|-----------|------|
| Trinet（トラスト＋SHIFT PLUS） | LoGoAIアシスタント | 自治体シェアNo.1、LGWAN対応 |
| エクサウィザーズ | 自治体AI zevo | LGWAN対応、自治体特化 |
| exaBase | exaBase 生成AI | マルチLLM対応 |
| ミンクスプラス | GaiXer | LGWAN対応、RAG内蔵 |
| Microsoft | Copilot for M365 | M365既存契約との連携 |
| ― | 公務員専用ChatGPTマサルくん | LGWAN対応 |

---

### 類型 II: AIチャットボット構築

**自治体が求めているもの**:
- 住民向けのFAQ応答・手続き案内チャットBot
- 多言語対応（インバウンド・外国人住民向け）
- LINE・Web・電話（ボイスBot）との連携
- 既存FAQデータの取り込み・更新運用
- 従来型チャットボットから生成AI対応へのリプレース

**典型的な公募要件**:
- 「AI総合案内サービスの構築・運用業務」
- 「チャットボットシステムの更新業務」（生成AI対応化）
- SaaSベースで構築し、運用も含めた年額契約が主流

**主要な競合・既存プレイヤー**:

| プレイヤー | 件数 | 特徴 |
|-----------|:---:|------|
| ビースポーク | 19件 | 観光・福祉特化、勝率100% |
| サイネックス | 19件 | 小規模自治体向け、低価格 |
| NTT東日本 | 数件 | 札幌市（9言語対応Bot）等 |

---

### 類型 III: RAG・ナレッジ検索構築

**自治体が求めているもの**:
- 庁内文書（条例・規則・マニュアル・通達）を検索できるAI
- 問い合わせ対応の効率化（住民向け・庁内向け）
- 議事録・答弁検索システム
- 既存文書管理システムとの連携

**典型的な公募要件**:
- 「庁内文書を活用したAI検索システムの構築」
- 「RAGを用いた問い合わせ対応システムの導入」
- 毎年約3倍のペースで案件数が増加中（1→3→10件）

**特徴**:
- SaaSプロダクト「だけ」では対応できず、自治体固有の文書に対するチューニングが必要
- データ整備（文書の前処理、チャンク設計、メタデータ付与）が工数の大半
- **NTTDXPNの技術力が最も活きる領域**

---

### 類型 IV: 業務特化AI開発

**自治体が求めているもの**:
- 議事録自動要約・文字起こし
- 申請書類の自動チェック・審査支援
- 保育所入所選考の自動化（さいたま市型）
- 福祉・子育て相談の面談支援AI
- 帳票データ入力の自動化
- 報告書・起案文書の自動生成

**典型的な公募要件**:
- 「AI議事録作成支援システムの導入」
- 「生成AIを活用した業務効率化ツールの開発」
- 業務ごとに要件が異なり、カスタム開発が前提

**特徴**:
- 既製品SaaSでは対応不可 → **スクラッチ開発が主戦場**
- 業務理解＋技術力の両方が求められる
- 単価が高くなりやすく、利益率も高い
- n8n + LLM API + 既存システム連携の組み合わせが有効

---

### 類型 V: AI研修・人材育成

**自治体が求めているもの**:
- 全職員向けの生成AIリテラシー研修
- 管理職向けのAI戦略セミナー
- プロンプトエンジニアリング実践研修
- AI活用推進リーダー育成
- 研修後のフォローアップ・定着支援

**典型的な公募要件**:
- 「生成AI活用に関する職員研修業務委託」
- 「AI利活用推進のための研修及び伴走支援業務」
- 研修単体 or AI環境導入とセットで発注

**特徴**:
- SaaS不要、人材とコンテンツで勝負
- 入口案件（研修→PoC→構築のアップセル導線）として戦略的に重要
- AI法施行（2025年9月）により自治体の「AI活用」が責務化 → 需要増

---

### 類型 VI: AI戦略・ロードマップ策定

**自治体が求めているもの**:
- AI活用ロードマップの策定
- ユースケース設計・優先度付け
- 業務棚卸し・AI適用性診断
- 第2世代交付金など補助金の申請支援
- CAIO設置に伴うAI戦略策定支援

**典型的な公募要件**:
- 「AI活用推進計画策定支援業務」
- 「DX推進に向けたAI活用調査及びロードマップ策定業務」
- DS-920ガイドライン対応のための計画策定

**特徴**:
- 件数は少ないが高単価になり得る
- 後続案件（PoC→構築→運用）への入口
- コンサル力・提案力で勝負

## 3. NTTDXPNの提案の4形態

NTTDXPNが自治体に対して取り得る提案形態を4つに分類する。

| 形態 | 概要 | 使う技術・リソース | 利益率 |
|------|------|----------------|:---:|
| **α: SaaS担ぎ + SI** | 既存SaaSプロダクトをパートナーとして担ぎ、導入・カスタマイズ・研修をセットで提供 | SaaS（exaBase, GaiXer等）+ SI | 低〜中 |
| **β: スクラッチ構築** | LLM API + Dify/n8n等を使い、自治体固有の要件に合わせてゼロから構築 | Dify, n8n, LLM API, Azure OpenAI, AWS Bedrock | **高** |
| **γ: コンサル/研修** | 人が価値を届ける。研修、伴走支援、戦略策定、効果検証 | 人材、コンテンツ、方法論 | **高** |
| **δ: ハイブリッド** | SaaS基盤の上に、スクラッチで付加価値を載せる（SaaS + RAG追加構築等） | SaaS + API + Dify/n8n | 中〜高 |

### 各形態の詳細

#### α: SaaS担ぎ + SI

```
[SaaSベンダー] → [DXPN: 導入・設定・研修・運用] → [自治体]
```

- 自治体が求める「プロダクト」をSaaSベンダーから調達し、DXPNがSIとして導入・運用を担う
- **メリット**: 応札できる案件の幅が広がる。自治体の「セキュリティ担保された既製品がほしい」というニーズに応えられる
- **デメリット**: SaaS部分は薄利。パートナー契約が必要。差別化が難しい
- **収益構造（3,000万円案件の場合）**:

| 項目 | 金額 | DXPNの利益 |
|------|-----:|:---:|
| SaaSライセンス | 1,200〜1,500万円 | △（ベンダー売上） |
| 導入・設定・カスタマイズ | 500〜800万円 | ◎ |
| 研修・定着支援 | 200〜300万円 | ◎ |
| 運用保守（年額） | 300〜500万円 | ◎（ストック型） |

#### β: スクラッチ構築

```
[LLM API] → [DXPN: 設計・開発・テスト・運用] → [自治体]
```

- LLM APIやOSSプラットフォーム（Dify, n8n等）を使い、自治体の要件に合わせてカスタム構築
- **メリット**: 全額がDXPNの売上。技術力で差別化。利益率が高い
- **デメリット**: 工数リスク。自治体に「プロダクト」が見えにくく、提案の説得力を作るのに工夫が必要
- **適用案件**: RAG構築、業務特化AI、業務自動化

#### γ: コンサル/研修

```
[DXPN: 人・知識・方法論] → [自治体]
```

- プロダクトもSIも不要。人と知識で価値を提供
- **メリット**: 低コストで始められる。顧客接点の入口として最適。利益率が高い
- **デメリット**: 単体では単価が小さい。スケールしにくい
- **戦略的位置付け**: 研修→PoC→構築→運用のアップセル入口

#### δ: ハイブリッド（SaaS + スクラッチ付加価値）

```
[SaaS基盤] + [DXPN: RAG追加構築 / 業務特化カスタマイズ] → [自治体]
```

- SaaSで全庁AI環境を整えつつ、DXPNの技術力でRAGや業務特化機能を追加構築
- **メリット**: 汎用ニーズ（AI環境）と特化ニーズ（業務AI）を同時に満たせる。SaaSベンダーとの競合を回避し、補完関係を作れる
- **デメリット**: 2社分の調整コスト。提案書の構成が複雑になりがち
- **これが最も現実的な戦い方** — 第1波（汎用）に乗りつつ、第2波（特化）でDXPN固有の価値を発揮する

## 4. 案件類型 × 提案形態マトリクス

**各セルの記号**: ◎ = 最適 / ○ = 有効 / △ = 可能だが不利 / × = 非対応

| 案件類型 | α: SaaS担ぎ+SI | β: スクラッチ | γ: コンサル/研修 | δ: ハイブリッド | **推奨戦略** |
|---------|:---:|:---:|:---:|:---:|------|
| **I: 全庁AI環境** | **◎** | △ | ○（付帯） | ◎ | SaaSを担いで導入SI。研修をセットで |
| **II: チャットボット** | **◎** | △ | △ | ○ | SaaSを担ぐ。ただし既存勢が強く差別化が課題 |
| **III: RAG構築** | △ | **◎** | ○（付帯） | **◎** | スクラッチが最適。SaaS＋RAG追加も有効 |
| **IV: 業務特化AI** | × | **◎** | ○（付帯） | ○ | スクラッチ一択。DXPNの主戦場 |
| **V: AI研修** | × | × | **◎** | × | コンサル/研修のみ。入口案件として戦略活用 |
| **VI: 戦略策定** | × | × | **◎** | × | コンサルのみ。後続案件への導線 |

---

## 5. 案件類型ごとの提案方針

### 類型 I: 全庁AI環境導入 → α（SaaS担ぎ+SI）or δ（ハイブリッド）

**課題**: SaaS商材がなければ応札すらできないケースが多い

**方針**:
1. **SaaSパートナー契約を締結する**（候補: exaBase, GaiXer, LoGoAI等）
2. SaaSライセンス + 導入支援 + 研修 + 運用をパッケージ化
3. δ型として、SaaS導入に加えRAG等の付加価値構築を提案に含める
4. 研修（γ）を厚めに設計し、「導入して終わり」ではなく「定着まで伴走」を差別化要素にする

**NTTグループ連携**: NTT東日本の生成AIサービス（2025年4月〜）との連携も選択肢。NTT東がSaaS提供、DXPNがSI・研修を担う分業モデル。

**担ぐべきSaaSの選定基準**:

| 基準 | 重要度 | 理由 |
|------|:---:|------|
| LGWAN対応 | **必須** | 自治体案件の大前提 |
| マルチLLM対応 | 高 | LLM切り替え要望が増加中 |
| RAG機能内蔵 or API拡張性 | 高 | δ型でDXPNが付加価値を載せるため |
| パートナー制度の利益率 | 中 | SaaS部分の利益率を確認 |
| 実績・認知度 | 中 | 自治体の安心感に直結 |

---

### 類型 II: AIチャットボット構築 → α（SaaS担ぎ）

**課題**: ビースポーク（勝率100%）・サイネックス等の既存SaaS勢が強い

**方針**:
1. **正面勝負は避ける** — チャットボット専業SaaSとの価格・機能競争は不利
2. 応札するなら、既存SaaS（ビースポーク、サイネックス等）をDXPNが担いで提案
3. **差別化ポイント**: 生成AI対応へのリプレース提案（従来型→生成AI型）。多言語対応。LINEとの連携。
4. チャットボット単体ではなく、「全庁AI環境（I型）+ 住民向けBot（II型）」のバンドル提案が有効

**優先度**: ★★☆（主力にはせず、I型・III型・IV型と組み合わせて提案）

---

### 類型 III: RAG・ナレッジ検索構築 → β（スクラッチ）or δ（ハイブリッド）

**課題**: 自治体固有の文書に対するチューニングが成功の鍵

**方針**:
1. **DXPNの技術力が最も活きる領域** — 積極的に狙う
2. Dify + LLM API + ベクトルDB でRAGを構築（β型）
3. 全庁AI環境（I型）の上にRAG機能を追加する提案も有効（δ型）
4. データ整備（文書の前処理、チャンク設計、メタデータ付与）をサービスメニューとして確立

**技術スタック**:

| コンポーネント | 選択肢 |
|-------------|--------|
| LLM | Azure OpenAI / AWS Bedrock / tsuzumi 2 |
| オーケストレーション | Dify |
| ベクトルDB | Qdrant / pgvector / Pinecone |
| 文書前処理 | LlamaParse / Unstructured |

**優先度**: ★★★（最注力。毎年3倍成長中）

---

### 類型 IV: 業務特化AI開発 → β（スクラッチ）

**課題**: 業務理解が必要。自治体側の業務フローの可視化から始める必要がある

**方針**:
1. **DXPNの独自領域** — SaaSベンダーがカバーできない
2. n8n + LLM API + UiPath で業務プロセスにAIを組み込む（C型: AI自動化）
3. 「人がAIを使う」（B型）だけでなく、「AIがプロセスの中で自動的に動く」（C型）を提案に含める
4. 議事録要約、申請自動チェック、報告書自動生成 等、横展開しやすいテーマから攻める

**横展開可能なユースケース**:

| ユースケース | 技術 | 単価帯 |
|------------|------|-----:|
| 議事録自動要約 | 音声→テキスト→LLM要約→配布自動化 | 300〜800万円 |
| 申請書類自動チェック | OCR→LLM判定→修正指示 | 500〜1,500万円 |
| 報告書・起案文書生成 | テンプレート+LLM生成+品質チェック | 300〜1,000万円 |
| 問い合わせ対応自動化 | RAG+分類→自動回答/担当振分 | 500〜1,500万円 |
| データ分析レポート自動化 | CSV取込→LLM分析→ダッシュボード生成 | 300〜800万円 |

**優先度**: ★★★（利益率が高く、DXPNの技術差別化が効く。第2波の本命）

---

### 類型 V: AI研修・人材育成 → γ（コンサル/研修）

**課題**: 単体では単価が小さい

**方針**:
1. **戦略的入口案件**として位置付ける（研修→PoC→構築のアップセル導線）
2. 3種の研修パッケージを用意:
   - A: 経営層・管理職向け（AI戦略セミナー、半日、20〜30万円）
   - B: 全職員向け（プロンプト実践研修、1日、30〜50万円）
   - C: 推進担当向け（PoC設計・技術基礎、1〜2日、50〜80万円）
3. 研修の中で業務課題をヒアリングし、PoC提案につなげる
4. AI法施行を追い風に「AI活用は自治体の責務」というメッセージで需要喚起

**優先度**: ★★☆（入口として重要。単体収益より後続案件獲得を重視）

---

### 類型 VI: AI戦略・ロードマップ策定 → γ（コンサル）

**課題**: 件数が少なく、コンサルファームとの競合もあり得る

**方針**:
1. 後続案件（PoC→構築→運用）への導線として活用
2. 第2世代交付金の申請支援を組み込み、「予算確保も含めてサポート」を訴求
3. DS-920ガイドライン対応・CAIO設置支援をフックに提案
4. NTTグループの信頼性を活かし、コンサルファームとの差別化は「実行力」で行う

**優先度**: ★★☆（件数は少ないが、大型案件への入口になる）

## 6. 総合戦略サマリー

### DXPNの注力ポートフォリオ

```
                    SaaS必要度 高 ←――――――――→ SaaS必要度 低

利益率 高    │  δ: ハイブリッド    │  β: スクラッチ構築    │
             │  (I+III型の複合)     │  (III, IV型)          │
             │                      │  ★ DXPNの主戦場      │
             │──────────────────────│──────────────────────│
利益率 低    │  α: SaaS担ぎ+SI    │  γ: コンサル/研修     │
             │  (I, II型)          │  (V, VI型)            │
             │  案件数は多い        │  ★ 入口案件          │
```

### 時間軸で見た戦略

| 時間軸 | 主な案件類型 | 提案形態 | アクション |
|--------|-----------|---------|-----------|
| **即時（0〜3ヶ月）** | V: 研修、VI: 戦略策定 | γ | 研修パッケージ即提案。入札キングで案件モニタリング |
| **短期（3〜6ヶ月）** | I: 全庁AI環境、III: RAG | α, δ | SaaSパートナー契約締結。RAG提案テンプレ整備 |
| **中期（6〜12ヶ月）** | IV: 業務特化AI | β | 横展開可能なユースケースで実績を作る |
| **長期（12ヶ月〜）** | 全類型の運用・定着 | δ, γ | ストック収益の積み上げ。源内展開後のカスタマイズ需要 |

### 案件獲得チャネルとの掛け合わせ

| チャネル | 向いている案件類型 | 理由 |
|---------|-----------------|------|
| **入札キングモニタリング** | I, II, III, IV | 公募案件を直接検知。Medium帯が主戦場 |
| **NTT東 支店ロビー活動** | V → I, III | 研修で入口を作り、環境導入・RAGにアップセル |
| **直接アプローチ** | IV, VI | 不調案件への再公募対応。既存顧客への提案 |
