# 自治体案件の種類 × NTTDXPNの提案の種類

> **目的**: 自治体の生成AI関連案件を種類別に分類し、それぞれに対してNTTDXパートナーのAIチームがどのような提案形態で臨めるかを整理する。
> **背景**: NTTDXPNは自前の生成AI SaaSプロダクトを持たないため、案件種類ごとに「既存SaaSを担ぐ」「スクラッチ開発」「コンサル/研修」の使い分けが必要。
> **日付**: 2026-03-02

---

## 1. 前提: NTTDXPNの構造的制約

NTTDXPNのAIチームは以下の特性を持つ。この制約を前提に、案件種類ごとの戦い方を整理する。

| 項目 | 状況 |
|------|------|
| **自社SaaS** | なし（生成AIのチャット基盤SaaSを持っていない） |
| **技術力** | LLM API活用、RAG構築、Dify/n8n/UiPathでの業務自動化に強み |
| **提供可能範囲** | SI（設計・構築・運用）、コンサル、研修 |
| **NTTグループ連携** | NTT東日本の支店ネットワーク、tsuzumi等のグループ資産を活用可能 |

### バリューチェーン上の位置

```
LLMベンダー → SaaSベンダー → SI/導入支援   → 自治体
(OpenAI等)    (exaBase等)     (★ここがDXPN)    (エンド)
```

**自治体が購入する3つのレイヤー**:

| レイヤー | 内容 | DXPNの対応力 |
|----------|------|:---:|
| (1) プロダクト | AI環境そのもの（SaaSライセンス） | △ 担ぐ必要あり |
| (2) フィッティング | 組織に合わせた構築（RAG、業務設計、連携） | **◎** |
| (3) 定着 | 研修、伴走支援、効果検証 | **◎** |

## 2. 自治体案件の6類型

入札王データ（887件）および公募調査から、自治体の生成AI関連案件を以下の6類型に分類する。

### 類型一覧

| # | 案件類型 | 概要 | 年間件数（推計） | 単価帯 | SaaS必要度 |
|---|---------|------|:---:|------:|:---:|
| **I** | **全庁AI環境導入** | 全職員が使える生成AIチャット環境の構築 | 80〜100件 | 300〜3,000万円 | **高** |
| **II** | **AIチャットボット構築** | 住民向けFAQ・問い合わせ対応Bot | 40〜50件 | 200〜1,000万円 | **高** |
| **III** | **RAG・ナレッジ検索構築** | 庁内文書・条例・マニュアルの検索システム | 10〜20件 | 500〜2,000万円 | **中** |
| **IV** | **業務特化AI開発** | 特定業務向けAI（議事録要約、申請処理等） | 20〜30件 | 300〜2,000万円 | **低** |
| **V** | **AI研修・人材育成** | 職員向けAIリテラシー研修、ワークショップ | 40〜50件 | 30〜500万円 | **なし** |
| **VI** | **AI戦略・ロードマップ策定** | AI活用計画策定、PoC設計、交付金申請支援 | 10〜15件 | 200〜800万円 | **なし** |

---

### 類型 I: 全庁AI環境導入

**自治体が求めているもの**:
- 全職員がセキュアに生成AIを使えるチャット環境
- LGWAN対応またはα'モデル準拠のセキュリティ
- 利用ログの管理・監査機能
- 簡単なUI（プロンプト不要の「モード機能」等）
- ガイドライン整備・運用ルール策定を含む

**典型的な公募要件**:
- 「庁内で安全に利用できる生成AIツールの導入及び運用業務」
- SaaSライセンス＋導入支援＋研修がセット発注されることが多い
- 利用者数に応じた月額課金 or 年額一括

**主要な競合・既存プレイヤー**:

| プレイヤー | サービス名 | 特徴 |
|-----------|-----------|------|
| Trinet（トラスト＋SHIFT PLUS） | LoGoAIアシスタント | 自治体シェアNo.1、LGWAN対応 |
| エクサウィザーズ | 自治体AI zevo | LGWAN対応、自治体特化 |
| exaBase | exaBase 生成AI | マルチLLM対応 |
| ミンクスプラス | GaiXer | LGWAN対応、RAG内蔵 |
| Microsoft | Copilot for M365 | M365既存契約との連携 |
| ― | 公務員専用ChatGPTマサルくん | LGWAN対応 |

---

### 類型 II: AIチャットボット構築

**自治体が求めているもの**:
- 住民向けのFAQ応答・手続き案内チャットBot
- 多言語対応（インバウンド・外国人住民向け）
- LINE・Web・電話（ボイスBot）との連携
- 既存FAQデータの取り込み・更新運用
- 従来型チャットボットから生成AI対応へのリプレース

**典型的な公募要件**:
- 「AI総合案内サービスの構築・運用業務」
- 「チャットボットシステムの更新業務」（生成AI対応化）
- SaaSベースで構築し、運用も含めた年額契約が主流

**主要な競合・既存プレイヤー**:

| プレイヤー | 件数 | 特徴 |
|-----------|:---:|------|
| ビースポーク | 19件 | 観光・福祉特化、勝率100% |
| サイネックス | 19件 | 小規模自治体向け、低価格 |
| NTT東日本 | 数件 | 札幌市（9言語対応Bot）等 |

---

### 類型 III: RAG・ナレッジ検索構築

**自治体が求めているもの**:
- 庁内文書（条例・規則・マニュアル・通達）を検索できるAI
- 問い合わせ対応の効率化（住民向け・庁内向け）
- 議事録・答弁検索システム
- 既存文書管理システムとの連携

**典型的な公募要件**:
- 「庁内文書を活用したAI検索システムの構築」
- 「RAGを用いた問い合わせ対応システムの導入」
- 毎年約3倍のペースで案件数が増加中（1→3→10件）

**特徴**:
- SaaSプロダクト「だけ」では対応できず、自治体固有の文書に対するチューニングが必要
- データ整備（文書の前処理、チャンク設計、メタデータ付与）が工数の大半
- **NTTDXPNの技術力が最も活きる領域**

---

### 類型 IV: 業務特化AI開発

**自治体が求めているもの**:
- 議事録自動要約・文字起こし
- 申請書類の自動チェック・審査支援
- 保育所入所選考の自動化（さいたま市型）
- 福祉・子育て相談の面談支援AI
- 帳票データ入力の自動化
- 報告書・起案文書の自動生成

**典型的な公募要件**:
- 「AI議事録作成支援システムの導入」
- 「生成AIを活用した業務効率化ツールの開発」
- 業務ごとに要件が異なり、カスタム開発が前提

**特徴**:
- 既製品SaaSでは対応不可 → **スクラッチ開発が主戦場**
- 業務理解＋技術力の両方が求められる
- 単価が高くなりやすく、利益率も高い
- n8n + LLM API + 既存システム連携の組み合わせが有効

---

### 類型 V: AI研修・人材育成

**自治体が求めているもの**:
- 全職員向けの生成AIリテラシー研修
- 管理職向けのAI戦略セミナー
- プロンプトエンジニアリング実践研修
- AI活用推進リーダー育成
- 研修後のフォローアップ・定着支援

**典型的な公募要件**:
- 「生成AI活用に関する職員研修業務委託」
- 「AI利活用推進のための研修及び伴走支援業務」
- 研修単体 or AI環境導入とセットで発注

**特徴**:
- SaaS不要、人材とコンテンツで勝負
- 入口案件（研修→PoC→構築のアップセル導線）として戦略的に重要
- AI法施行（2025年9月）により自治体の「AI活用」が責務化 → 需要増

---

### 類型 VI: AI戦略・ロードマップ策定

**自治体が求めているもの**:
- AI活用ロードマップの策定
- ユースケース設計・優先度付け
- 業務棚卸し・AI適用性診断
- 第2世代交付金など補助金の申請支援
- CAIO設置に伴うAI戦略策定支援

**典型的な公募要件**:
- 「AI活用推進計画策定支援業務」
- 「DX推進に向けたAI活用調査及びロードマップ策定業務」
- DS-920ガイドライン対応のための計画策定

**特徴**:
- 件数は少ないが高単価になり得る
- 後続案件（PoC→構築→運用）への入口
- コンサル力・提案力で勝負

## 3. NTTDXPNの提案の4形態

NTTDXPNが自治体に対して取り得る提案形態を4つに分類する。

| 形態 | 概要 | 使う技術・リソース | 利益率 |
|------|------|----------------|:---:|
| **α: SaaS担ぎ + SI** | 既存SaaSプロダクトをパートナーとして担ぎ、導入・カスタマイズ・研修をセットで提供 | SaaS（exaBase, GaiXer等）+ SI | 低〜中 |
| **β: スクラッチ構築** | LLM API + Dify/n8n等を使い、自治体固有の要件に合わせてゼロから構築 | Dify, n8n, LLM API, Azure OpenAI, AWS Bedrock | **高** |
| **γ: コンサル/研修** | 人が価値を届ける。研修、伴走支援、戦略策定、効果検証 | 人材、コンテンツ、方法論 | **高** |
| **δ: ハイブリッド** | SaaS基盤の上に、スクラッチで付加価値を載せる（SaaS + RAG追加構築等） | SaaS + API + Dify/n8n | 中〜高 |

### 各形態の詳細

#### α: SaaS担ぎ + SI

既存SaaSプロダクトを活用し、DXPNがSIとして導入・運用を担う形態。SaaSの契約形態によって以下の4パターンに分かれる。

##### SaaSの契約形態パターン

| # | パターン | 契約の流れ | パートナー契約 | DXPNの位置付け |
|---|---------|-----------|:---:|------------|
| **α-1** | **分離発注** | 自治体がSaaSベンダーと直接契約。DXPNはSI部分のみ受注 | **不要** | 導入支援・研修・運用の受託者 |
| **α-2** | **ライセンス代行** | DXPNがSaaSを自ら契約し、利用料込みで自治体に一括提供 | **不要**（通常の利用契約でOK） | サービス提供者（ライセンス込み） |
| **α-3** | **再販** | DXPNが正規再販パートナーとしてSaaSライセンスを販売 | **必要** | 再販パートナー |
| **α-4** | **共同提案** | SaaSベンダーとDXPNが共同で応札（JV的） | 案件単位の合意 | 共同提案者 |

##### 各パターンの詳細

**α-1: 分離発注**
```
自治体 ──契約①──→ SaaSベンダー（ライセンス）
自治体 ──契約②──→ DXPN（導入支援・研修・運用）
```
- 自治体が公募を「ツール調達」と「導入支援」で分けて発注する場合
- DXPNはSI部分だけに応札すればよく、パートナー契約は不要
- 大規模自治体で比較的多いパターン
- **メリット**: SaaSベンダーとの契約関係が不要。SI部分の利益率が高い
- **デメリット**: 案件の中で「ツール選定」に関与しにくい。自治体が別途選んだSaaSに合わせる必要がある

**α-2: ライセンス代行（収納代行型）** ← 実際の自治体案件で多いパターン
```
SaaSベンダー ──利用契約──→ DXPN ──一括契約──→ 自治体
                            （ライセンス料 + SI費用を一本で請求）
```
- DXPNがSaaSを自ら利用契約し、自治体にはDXPNとの一本の契約でライセンス込みのサービスを提供
- 自治体にとっては「DXPNに発注すれば全部やってくれる」状態になる
- **SaaSの正規再販パートナーである必要はない** — DXPNが通常のユーザーとしてSaaSを契約し、そのコストを自治体への提供費用に含める
- **実際の自治体案件ではこの形態が多い** — 自治体は調達の手間を減らしたい（1社との契約で済む）
- **メリット**: 一括発注の公募に応札できる。自治体に「窓口はDXPN一本」の安心感を提供。パートナー契約不要
- **デメリット**: SaaSコストの価格転嫁・利益設計が必要。SaaSベンダーの価格変更リスクを負う

**α-3: 再販**
```
SaaSベンダー ──再販契約──→ DXPN ──再販──→ 自治体
```
- 正規のパートナープログラムに参加し、再販権を取得して販売
- **メリット**: パートナー価格（卸値）で仕入れられる。共同マーケティング・リード紹介の可能性
- **デメリット**: パートナー契約の締結が必要。販売目標・認定要件がある場合も

**α-4: 共同提案**
```
[SaaSベンダー + DXPN] ──共同応札──→ 自治体
```
- 案件単位でSaaSベンダーと協業合意を結び、共同で応札
- **メリット**: 案件ごとに柔軟に組み合わせ可能。パートナープログラムに縛られない
- **デメリット**: 案件ごとの調整コスト。継続性が担保されにくい

##### 収益構造の比較（3,000万円案件の場合）

| 項目 | α-1: 分離発注 | α-2: ライセンス代行 | α-3: 再販 |
|------|-----:|-----:|-----:|
| SaaSライセンス | **対象外** | 1,200〜1,500万円（原価） | 1,000〜1,300万円（卸値） |
| ライセンス利益 | — | 0〜300万円（上乗せ分） | 200〜500万円（再販マージン） |
| DXPNの受注額 | 1,500〜1,800万円（SI部分のみ） | **3,000万円（全額）** | **3,000万円（全額）** |
| SI部分の利益 | ◎ | ◎ | ◎ |
| 合計利益率 | 高（SI部分100%） | 中〜高（全体の50〜60%） | 中〜高（マージン次第） |

##### どのパターンを使うべきか

| 判断基準 | 推奨パターン |
|---------|-----------|
| 自治体が一括発注を希望 / 公募が一括 | **α-2（ライセンス代行）** が最も現実的 |
| 自治体が分離発注する場合 | **α-1（分離発注）** でSI部分に集中 |
| SaaSベンダーと継続的に組みたい | **α-3（再販）** で関係構築 |
| 初めてのSaaSベンダーとの協業 | **α-4（共同提案）** でまず1件やってみる |
| とにかく早く応札したい | **α-2（ライセンス代行）** — パートナー契約不要ですぐ動ける |

#### β: スクラッチ構築

```
[LLM API] → [DXPN: 設計・開発・テスト・運用] → [自治体]
```

- LLM APIやOSSプラットフォーム（Dify, n8n等）を使い、自治体の要件に合わせてカスタム構築
- **メリット**: 全額がDXPNの売上。技術力で差別化。利益率が高い
- **デメリット**: 工数リスク。自治体に「プロダクト」が見えにくく、提案の説得力を作るのに工夫が必要
- **適用案件**: RAG構築、業務特化AI、業務自動化

#### γ: コンサル/研修

```
[DXPN: 人・知識・方法論] → [自治体]
```

- プロダクトもSIも不要。人と知識で価値を提供
- **メリット**: 低コストで始められる。顧客接点の入口として最適。利益率が高い
- **デメリット**: 単体では単価が小さい。スケールしにくい
- **戦略的位置付け**: 研修→PoC→構築→運用のアップセル入口

#### δ: ハイブリッド（SaaS + スクラッチ付加価値）

```
[SaaS基盤] + [DXPN: RAG追加構築 / 業務特化カスタマイズ] → [自治体]
```

- SaaSで全庁AI環境を整えつつ、DXPNの技術力でRAGや業務特化機能を追加構築
- **メリット**: 汎用ニーズ（AI環境）と特化ニーズ（業務AI）を同時に満たせる。SaaSベンダーとの競合を回避し、補完関係を作れる
- **デメリット**: 2社分の調整コスト。提案書の構成が複雑になりがち
- **これが最も現実的な戦い方** — 第1波（汎用）に乗りつつ、第2波（特化）でDXPN固有の価値を発揮する

### 技術インフラ・開発ツールの全体像

提案形態ごとに、「どのインフラの上で」「何を使って」構築するかのレイヤー構造を整理する。

#### レイヤー構造

```
┌─────────────────────────────────────────────────────────┐
│ [L4] アプリケーション / ユーザー接点                        │
│   SaaS製品(exaBase等) / Webアプリ / LINE Bot / Teams連携   │
├─────────────────────────────────────────────────────────┤
│ [L3] 開発フレームワーク・ツール                             │
│   Dify / n8n / LangChain / FastAPI / UiPath              │
├─────────────────────────────────────────────────────────┤
│ [L2] AI基盤 / LLM API                                    │
│   Azure OpenAI / AWS Bedrock / Vertex AI / tsuzumi       │
├─────────────────────────────────────────────────────────┤
│ [L1] インフラ基盤                                         │
│   AWS / Azure / GCP / NTT DC(プライベートクラウド)         │
└─────────────────────────────────────────────────────────┘
```

#### L1: インフラ基盤の選択肢

| インフラ | 特徴 | LGWAN接続 | 自治体案件での適性 |
|---------|------|:---:|---------|
| **AWS** | サービス豊富。Bedrock直結。自治体の実績も増加中 | △（構成次第） | ◎ スクラッチ・RAGに強い |
| **Azure** | Azure OpenAI直結。自治体のMS依存率が高い（M365等）。政府クラウド認定 | ○ | ◎ 自治体親和性が最も高い |
| **GCP** | Vertex AI / Gemini。コスト最適化しやすい | △ | ○ Gemini活用案件 |
| **NTT DC / プライベートクラウド** | NTTグループ資産。閉域網対応。LGWAN-ASP対応 | **◎** | ◎ セキュリティ最重視案件 |
| **LGWAN-ASP** | LGWAN上でサービス提供する場合の専用基盤 | **◎（必須）** | ◎ LGWAN系統からの利用 |

**自治体案件でのインフラ選定の判断基準**:

| 判断基準 | 推奨インフラ |
|---------|-----------|
| LGWAN系統から利用する | NTT DC / LGWAN-ASP |
| インターネット系統から利用する | Azure（MS親和性）/ AWS |
| NTTグループ提案として | NTT DC + tsuzumi（グループシナジー訴求） |
| コスト最適化重視 | AWS / GCP |
| 自治体がMS365を利用中 | Azure（既存契約との統合） |

#### L2: LLM API基盤の選択肢

| LLM API基盤 | 提供元 | 動くインフラ | 特徴 |
|------------|--------|-----------|------|
| **Azure OpenAI Service** | Microsoft/OpenAI | Azure | GPT-4o, GPT-4.1。自治体のAzure利用率が高く親和性◎。データ残留なし保証 |
| **AWS Bedrock** | Amazon | AWS | Claude, Titan, Llama等マルチモデル。モデル切替が容易 |
| **Google Vertex AI** | Google | GCP | Gemini。マルチモーダル（画像・PDF解析）に強い |
| **tsuzumi** | NTT | NTT DC / Azure | 日本語特化。NTTグループシナジー。行政文書への適合度検証が進行中 |
| **セルフホストOSS LLM** | 各社 | 任意 | Llama 3, Mistral等をNTT DC上に自前デプロイ。完全閉域対応 |

#### L3: 開発フレームワーク・ツール

##### ローコード系

| ツール | 種別 | SaaS版 | セルフホスト版 | 適用領域 |
|--------|------|:---:|:---:|---------|
| **Dify** | LLMオーケストレーション | ○（Dify Cloud） | **◎**（推奨） | RAG構築、チャットBot、ワークフロー |
| **n8n** | ワークフロー自動化 | ○（n8n Cloud） | **◎**（推奨） | 業務自動化、API連携、定期バッチ |
| **UiPath** | RPA + AI | ○（UiPath Cloud） | ○ | 既存業務のRPA化 + AI連携 |

**セルフホスト版の構成イメージ（Difyの場合）**:
```
[Azure / AWS / NTT DC]
  └─ VM or Kubernetes (EKS / AKS)
       ├─ Dify (Docker)
       ├─ PostgreSQL (メタデータ)
       ├─ Redis (キャッシュ)
       └─ Qdrant / pgvector (ベクトルDB)
           ↓ API呼出
       [Azure OpenAI / Bedrock / tsuzumi]
```
- DXPNがインフラを選定し、Difyをデプロイ・設定する
- SaaS版は手軽だが、**自治体案件ではセルフホスト版が基本**（データ主権・閉域網要件）
- セルフホスト版なら**インフラの選択権はDXPNにある** → NTT DC上に構築も可能

##### コードベース（フルスクラッチ）

| 言語・フレームワーク | 用途 | 適性 |
|-------------------|------|------|
| **Python + FastAPI** | AIバックエンドAPI開発 | ◎ LLM連携の標準。LangChain/LlamaIndex等のエコシステムが充実 |
| **Python + LangChain** | LLMオーケストレーション（コードベース） | ◎ RAGパイプライン、Agent構築。Difyより柔軟 |
| **Python + LlamaIndex** | RAG特化フレームワーク | ◎ 高度なRAG（マルチインデックス、再ランキング等） |
| **TypeScript + Next.js** | フロントエンド / フルスタック | ○ チャットUI、管理画面 |
| **Python + Streamlit** | プロトタイプ / 内部ツールUI | ○ PoC・検証用途に最適 |

**コードベースの構成イメージ**:
```
[Azure / AWS / NTT DC]
  └─ VM or Kubernetes or App Service
       ├─ FastAPI (Python) ← アプリケーション本体
       │    ├─ LangChain / LlamaIndex
       │    ├─ 業務ロジック
       │    └─ API連携 (LLM, DB, 外部システム)
       ├─ Next.js (フロントエンド)
       ├─ PostgreSQL + pgvector
       └─ Redis / Celery (非同期処理)
           ↓ API呼出
       [Azure OpenAI / Bedrock]
```
- Dify/n8nでは対応できない複雑な要件に適用
- 例: 複数システム連携、独自のUI/UX、高度なセキュリティ要件、既存業務システムとの深い統合

#### 提案形態ごとの技術構成マップ

| 形態 | L1: インフラ | L2: LLM API | L3: 開発ツール | L4: アプリ |
|------|-----------|------------|--------------|----------|
| **α: SaaS担ぎ** | SaaSベンダー管理（DXPNは関与なし） | SaaSに内包 | SaaSに内包 | SaaS製品そのもの |
| **β: スクラッチ（ローコード）** | **AWS / Azure / NTT DC** を選定・構築 | Azure OpenAI / Bedrock / tsuzumi | **Dify / n8n セルフホスト版** | Dify標準UI or カスタムUI |
| **β: スクラッチ（コードベース）** | **AWS / Azure / NTT DC** を選定・構築 | Azure OpenAI / Bedrock / tsuzumi | **Python + LangChain + FastAPI** | Next.js / Streamlit |
| **γ: コンサル/研修** | — | —（デモ用にSaaS利用程度） | — | — |
| **δ: ハイブリッド** | SaaS + **追加構築部分はAWS/Azure/NTT DC** | SaaS内包 + 追加部分はAzure OpenAI等 | 追加部分に**Dify or コードベース** | SaaS + 追加機能のUI |

#### ローコード vs コードベースの使い分け

| 判断基準 | ローコード（Dify/n8n） | コードベース（Python+FastAPI等） |
|---------|:---:|:---:|
| RAGチャットBot（標準的） | **◎** | ○ |
| 業務自動化ワークフロー | **◎** | ○ |
| 複数システム深い統合 | △ | **◎** |
| 独自UI/UX要件 | △ | **◎** |
| 高度なRAG（再ランキング、マルチインデックス等） | △ | **◎** |
| PoC・検証フェーズ | **◎** 短期で構築 | △ |
| 本番運用の保守性 | ○ | **◎** コードレビュー・テスト可能 |
| 開発スピード | **◎** | △ |
| DXPNのリソース効率 | **◎** 少人数で対応可能 | △ エンジニア工数大 |

**現実的な使い分け**:
- **PoC → Difyで素早く作る** → 本番化の判断後、要件次第でDifyのまま or コードベースに移行
- **標準的なRAG・チャットBot → Difyセルフホスト**で十分
- **業務自動化・API連携 → n8n**が適切
- **複雑な業務ロジック・独自UI・深いシステム連携 → コードベース**一択

## 4. 案件類型 × 提案形態マトリクス

**各セルの記号**: ◎ = 最適 / ○ = 有効 / △ = 可能だが不利 / × = 非対応

| 案件類型 | α: SaaS担ぎ+SI | β: スクラッチ | γ: コンサル/研修 | δ: ハイブリッド | **推奨戦略** |
|---------|:---:|:---:|:---:|:---:|------|
| **I: 全庁AI環境** | **◎** | △ | ○（付帯） | ◎ | SaaSを担いで導入SI。研修をセットで |
| **II: チャットボット** | **◎** | △ | △ | ○ | SaaSを担ぐ。ただし既存勢が強く差別化が課題 |
| **III: RAG構築** | △ | **◎** | ○（付帯） | **◎** | スクラッチが最適。SaaS＋RAG追加も有効 |
| **IV: 業務特化AI** | × | **◎** | ○（付帯） | ○ | スクラッチ一択。DXPNの主戦場 |
| **V: AI研修** | × | × | **◎** | × | コンサル/研修のみ。入口案件として戦略活用 |
| **VI: 戦略策定** | × | × | **◎** | × | コンサルのみ。後続案件への導線 |

## 5. 案件類型ごとの提案方針

### 類型 I: 全庁AI環境導入 → α（SaaS担ぎ+SI）or δ（ハイブリッド）

**課題**: 自前のSaaSプロダクトがないため、既存SaaSを何らかの形で調達する必要がある

**方針**:
1. **まずα-2（ライセンス代行）で始める** — パートナー契約不要で最もスピーディに動ける。DXPNがSaaSを自ら契約し、自治体には一括でサービス提供
2. 案件数が増えてきたら**α-3（再販パートナー）への移行を検討** — 卸値による利益率改善、ベンダーからのリード紹介等のメリット
3. 公募が分離発注の場合は**α-1（SI部分のみ）で応札** — SaaS調達は自治体に任せる
4. δ型として、SaaS導入に加えRAG等の付加価値構築を提案に含める
5. 研修（γ）を厚めに設計し、「導入して終わり」ではなく「定着まで伴走」を差別化要素にする

**契約形態の使い分け**:

| 公募の形態 | 推奨する契約パターン | 理由 |
|-----------|-----------------|------|
| 一括発注（ツール+SI+研修+運用） | **α-2: ライセンス代行** | パートナー契約不要で一括受注可能 |
| 分離発注（ツール調達は別途） | **α-1: 分離発注** | SI部分に集中。利益率高い |
| SaaSベンダー指定あり | **α-4: 共同提案** | 指定ベンダーと案件単位で協業 |

**NTTグループ連携**: NTT東日本の生成AIサービス（2025年4月〜）との連携も選択肢。NTT東がSaaS提供、DXPNがSI・研修を担う分業モデル（α-1型の社内版）。

**担ぐべきSaaSの選定基準**:

| 基準 | 重要度 | 理由 |
|------|:---:|------|
| LGWAN対応 | **必須** | 自治体案件の大前提 |
| マルチLLM対応 | 高 | LLM切り替え要望が増加中 |
| RAG機能内蔵 or API拡張性 | 高 | δ型でDXPNが付加価値を載せるため |
| ライセンス体系の柔軟性 | 高 | α-2（ライセンス代行）時の価格転嫁のしやすさ |
| 実績・認知度 | 中 | 自治体の安心感に直結 |

---

### 類型 II: AIチャットボット構築 → α（SaaS担ぎ）

**課題**: ビースポーク（勝率100%）・サイネックス等の既存SaaS勢が強い

**方針**:
1. **正面勝負は避ける** — チャットボット専業SaaSとの価格・機能競争は不利
2. 応札するなら、既存SaaS（ビースポーク、サイネックス等）をDXPNが担いで提案
3. **差別化ポイント**: 生成AI対応へのリプレース提案（従来型→生成AI型）。多言語対応。LINEとの連携。
4. チャットボット単体ではなく、「全庁AI環境（I型）+ 住民向けBot（II型）」のバンドル提案が有効

**優先度**: ★★☆（主力にはせず、I型・III型・IV型と組み合わせて提案）

---

### 類型 III: RAG・ナレッジ検索構築 → β（スクラッチ）or δ（ハイブリッド）

**課題**: 自治体固有の文書に対するチューニングが成功の鍵

**方針**:
1. **DXPNの技術力が最も活きる領域** — 積極的に狙う
2. Dify + LLM API + ベクトルDB でRAGを構築（β型）
3. 全庁AI環境（I型）の上にRAG機能を追加する提案も有効（δ型）
4. データ整備（文書の前処理、チャンク設計、メタデータ付与）をサービスメニューとして確立

**技術スタック**（→ 詳細は「技術インフラ・開発ツールの全体像」セクション参照）:

| レイヤー | 選択肢 | 備考 |
|---------|--------|------|
| L1: インフラ | Azure / AWS / NTT DC | Azure推奨（自治体のMS利用率高） |
| L2: LLM API | Azure OpenAI / Bedrock / tsuzumi | マルチモデル対応が望ましい |
| L3: 開発ツール | **Difyセルフホスト版**（標準） / LangChain+FastAPI（高度な要件） | PoCはDify → 本番も要件次第でDify継続 |
| ベクトルDB | Qdrant / pgvector / Pinecone | pgvectorはPostgreSQL統合で運用が楽 |
| 文書前処理 | LlamaParse / Unstructured | 自治体文書のPDF解析がカギ |

**優先度**: ★★★（最注力。毎年3倍成長中）

---

### 類型 IV: 業務特化AI開発 → β（スクラッチ）

**課題**: 業務理解が必要。自治体側の業務フローの可視化から始める必要がある

**方針**:
1. **DXPNの独自領域** — SaaSベンダーがカバーできない
2. n8n + LLM API + UiPath で業務プロセスにAIを組み込む（C型: AI自動化）
3. 「人がAIを使う」（B型）だけでなく、「AIがプロセスの中で自動的に動く」（C型）を提案に含める
4. 議事録要約、申請自動チェック、報告書自動生成 等、横展開しやすいテーマから攻める

**横展開可能なユースケース**:

| ユースケース | 技術 | 単価帯 |
|------------|------|-----:|
| 議事録自動要約 | 音声→テキスト→LLM要約→配布自動化 | 300〜800万円 |
| 申請書類自動チェック | OCR→LLM判定→修正指示 | 500〜1,500万円 |
| 報告書・起案文書生成 | テンプレート+LLM生成+品質チェック | 300〜1,000万円 |
| 問い合わせ対応自動化 | RAG+分類→自動回答/担当振分 | 500〜1,500万円 |
| データ分析レポート自動化 | CSV取込→LLM分析→ダッシュボード生成 | 300〜800万円 |

**優先度**: ★★★（利益率が高く、DXPNの技術差別化が効く。第2波の本命）

---

### 類型 V: AI研修・人材育成 → γ（コンサル/研修）

**課題**: 単体では単価が小さい

**方針**:
1. **戦略的入口案件**として位置付ける（研修→PoC→構築のアップセル導線）
2. 3種の研修パッケージを用意:
   - A: 経営層・管理職向け（AI戦略セミナー、半日、20〜30万円）
   - B: 全職員向け（プロンプト実践研修、1日、30〜50万円）
   - C: 推進担当向け（PoC設計・技術基礎、1〜2日、50〜80万円）
3. 研修の中で業務課題をヒアリングし、PoC提案につなげる
4. AI法施行を追い風に「AI活用は自治体の責務」というメッセージで需要喚起

**優先度**: ★★☆（入口として重要。単体収益より後続案件獲得を重視）

---

### 類型 VI: AI戦略・ロードマップ策定 → γ（コンサル）

**課題**: 件数が少なく、コンサルファームとの競合もあり得る

**方針**:
1. 後続案件（PoC→構築→運用）への導線として活用
2. 第2世代交付金の申請支援を組み込み、「予算確保も含めてサポート」を訴求
3. DS-920ガイドライン対応・CAIO設置支援をフックに提案
4. NTTグループの信頼性を活かし、コンサルファームとの差別化は「実行力」で行う

**優先度**: ★★☆（件数は少ないが、大型案件への入口になる）

## 6. 総合戦略サマリー

### DXPNの注力ポートフォリオ

```
                    SaaS必要度 高 ←――――――――→ SaaS必要度 低

利益率 高    │  δ: ハイブリッド    │  β: スクラッチ構築    │
             │  (I+III型の複合)     │  (III, IV型)          │
             │                      │  ★ DXPNの主戦場      │
             │──────────────────────│──────────────────────│
利益率 低    │  α: SaaS担ぎ+SI    │  γ: コンサル/研修     │
             │  (I, II型)          │  (V, VI型)            │
             │  案件数は多い        │  ★ 入口案件          │
```

### 時間軸で見た戦略

| 時間軸 | 主な案件類型 | 提案形態 | アクション |
|--------|-----------|---------|-----------|
| **即時（0〜3ヶ月）** | V: 研修、VI: 戦略策定 | γ | 研修パッケージ即提案。入札キングで案件モニタリング |
| **短期（3〜6ヶ月）** | I: 全庁AI環境、III: RAG | α, δ | SaaSパートナー契約締結。RAG提案テンプレ整備 |
| **中期（6〜12ヶ月）** | IV: 業務特化AI | β | 横展開可能なユースケースで実績を作る |
| **長期（12ヶ月〜）** | 全類型の運用・定着 | δ, γ | ストック収益の積み上げ。源内展開後のカスタマイズ需要 |

### 案件獲得チャネルとの掛け合わせ

| チャネル | 向いている案件類型 | 理由 |
|---------|-----------------|------|
| **入札キングモニタリング** | I, II, III, IV | 公募案件を直接検知。Medium帯が主戦場 |
| **NTT東 支店ロビー活動** | V → I, III | 研修で入口を作り、環境導入・RAGにアップセル |
| **直接アプローチ** | IV, VI | 不調案件への再公募対応。既存顧客への提案 |
