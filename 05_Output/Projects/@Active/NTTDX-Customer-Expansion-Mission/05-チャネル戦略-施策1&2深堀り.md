## 1. 戦略の前提整理

### 1.1 AIチームの立ち位置

NTTDXパートナー内のコンサルティング型チームとして、生成AI案件の企画〜実装〜定着までを一気通貫で支援する。少数精鋭のため、**案件の量より質**が重要。高単価・高付加価値の案件を安定的に獲得する仕組みが必要。

### 1.2 インバウンド主軸の理由

| 観点 | インバウンドが適する理由 |
|---|---|
| **NTTブランド** | 親会社NTT東の信頼・知名度を活用でき、コールドアプローチが不要 |
| **案件の性質** | 生成AI導入は信頼ベースの意思決定。紹介経由の方が成約率が高い |
| **チーム規模** | 少人数のため大量のアウトバウンド営業は非効率。質の高いリードに集中すべき |
| **既存ネットワーク** | NTT東の支店網・事業部ネットワークが既に存在する |

### 1.3 現状の課題

- **チャネル管理が属人的**: 紹介経路が個人の関係性に依存し、再現性がない
- **Web不在**: AIチーム独自のWebページがなく、オーガニックな問い合わせ導線がゼロ
- **リード育成パイプラインなし**: 見込み顧客の管理・ナーチャリングの仕組みがない
- **コンテンツ資産の不足**: 事例紹介や技術発信が体系化されていない

### 1.4 チャネル戦略の目的

> **属人的な紹介を、管理可能・測定可能・スケーラブルなインバウンドエンジンに変える**

## 3. 施策1深堀り -- NTT東連携チャネル戦略

### 3.1 チャネルA1: BI本部VP

**概要**: NTT東のBI本部VP（ビジネスイノベーション本部バイスプレジデント）は、マーケットに対して面的に展開するサービス・仕組みの企画を担っている。ここと連携することで、AIチームのサービスをNTT東の商材ラインナップに組み込める。

**アプローチ方針**:
- AIサービスをNTT東の商材として**コセル（co-sell）**できる形に整える
- VP本部が自社顧客に提案しやすい「パッケージ化された商材」を提供する

**具体的施策**:

| # | 施策 | 内容 |
|---|---|---|
| 1 | 共同サービスメニュー作成 | NTT東の既存サービスとAIサービスのセット提案を設計 |
| 2 | 定期パイプラインレビュー | 四半期ごとにVP本部とパイプライン共有会議を実施 |
| 3 | 成功事例の共有 | VP経由で獲得した案件の成果を共有し、追加紹介を促進 |
| 4 | 紹介資料の最適化 | VP本部の担当者が顧客に説明しやすい1枚サマリーを用意 |

**KPI**:
- VP経由の紹介件数: 月2件以上
- 紹介→初回面談の転換率: 70%以上
- VP経由案件の受注金額: 四半期1,000万円以上

---

### 3.2 チャネルA2: 産業基盤ビジネス部

**概要**: 南支店BI（産業基盤ビジネス部）はロイヤルカスタマーを持っている。既存の信頼関係がある顧客に対して、AIサービスをアドオン提案する。自治体向け・民需向けの両方が対象。

**アプローチ方針**:
- **アカウントベースド戦略**: 上位10〜20社を特定し、個社別にAI活用仮説を準備
- 自治体と民需で提案切り口を分ける

**自治体向けアプローチ**:

| # | 施策 | 内容 |
|---|---|---|
| 1 | 自治体向けAI活用事例集作成 | 住民サービス、内部業務効率化、データ活用の事例整理 |
| 2 | 公募連動提案 | 施策3（公募案件）と連携し、自治体が予算化しやすいタイミングで提案 |
| 3 | 自治体DX研修パッケージ | 首長・幹部向け＋職員向けの研修を自治体向けにカスタマイズ |

**民需向けアプローチ**:

| # | 施策 | 内容 |
|---|---|---|
| 1 | 業種別AI活用仮説シート | 製造・医療・農業・教育等の業種別に「AI活用あるある」を整理 |
| 2 | 顧客課題ヒアリング同行 | 産業基盤の営業に同行し、直接顧客課題を把握 |
| 3 | PoC成功→横展開プレイブック | 1社のPoC成功を同業種他社への横展開につなげる手順を整備 |

**KPI**:
- ターゲットアカウント接触数: 月3社以上
- 同行商談→案件化率: 30%以上
- 自治体/民需の案件バランス: 目標比率を設定

---

### 3.3 チャネルA3: 支店ネットワーク

**概要**: NTT東の各地域支店の営業担当者がAI案件の"目利き"ができるようにし、紹介を増やす。「Train the Trainer」モデル。

**アプローチ方針**:
- 支店営業にAIの初歩から教える必要はない
- **「こういう相談が来たらAIチームに紹介してください」** という判断基準を明確にする

**具体的施策**:

| # | 施策 | 内容 |
|---|---|---|
| 1 | AI案件発見チェックリスト | 顧客がこんな発言をしたら→AIチームに紹介、というシンプルなリスト |
| 2 | 支店向け勉強会（30分） | 短時間で「AIチームができること」をデモ込みで紹介 |
| 3 | 紹介フォーム整備 | Slack or Google Form等で簡単に紹介できる仕組み |
| 4 | 紹介実績フィードバック | 紹介してくれた支店に対して、案件の結果をフィードバック |
| 5 | パイロット支店で検証 | まず3支店で試し、効果を確認してから全支店に展開 |

**AI案件発見チェックリスト（例）**:

> お客様がこんなことを言ったら、AIチームに紹介してください：
> - 「ChatGPTを業務で使いたい」「AIで業務を効率化したい」
> - 「社内の文書検索を楽にしたい」「問い合わせ対応を自動化したい」
> - 「データは溜まっているが活用できていない」
> - 「他社はAIをどう使っているのか知りたい」
> - 「AIのセキュリティが心配」「ガバナンスをどうすればいいか」

**KPI**:
- 活性化支店数（1件以上紹介した支店の割合）: 30%以上
- 支店経由の紹介件数: 月5件以上
- 紹介→面談までのリードタイム: 5営業日以内

---

### 3.4 チャネルA4: 社内他事業部

**概要**: DXパートナー内の他チームや、NTTグループ会社からの横連携による案件紹介。

**アプローチ方針**:
- AIチームの存在と能力を社内に周知する
- 他チームが受けた案件の中で「AI要素がある部分」をAIチームが引き受ける

**具体的施策**:

| # | 施策 | 内容 |
|---|---|---|
| 1 | 社内Slack/Teams告知 | 月1回、AIチームの最新事例・成果を社内チャネルで発信 |
| 2 | ランチ＆ラーンセッション | 隔月で社内向けのカジュアルな勉強会を開催 |
| 3 | クロスセル案件テンプレート | 他チームの提案にAIサービスをアドオンする際のテンプレート |

**KPI**:
- 社内紹介件数: 月1件以上
- クロスセル案件の受注率: 50%以上

---

### 3.5 紹介資料の設計方針

ターゲット別に3種類の資料を用意する：

| 資料 | ターゲット | 目的 | ポイント |
|---|---|---|---|
| **エグゼクティブサマリー**（1枚） | VP本部幹部 | 予算確保・承認 | ROI・事業インパクト中心 |
| **営業向けガイド**（3〜5枚） | 支店営業担当 | 紹介判断・顧客への初回紹介 | チェックリスト型、「こういう相談→紹介」のフロー |
| **顧客向けサービス紹介**（10〜15枚） | エンド顧客 | サービス理解・検討開始 | 事例中心、フェーズ別メニュー、次のステップ明示 |

## 4. 施策2深堀り -- Webチャネル戦略

### 4.1 インバウンドフライホイール設計

従来のファネル（漏斗型）ではなく、フライホイール（循環型）で設計する：

```
      ┌─────────────────────────────────────┐
      │                                     │
  Attract（集客）                      Delight（満足）
  コンテンツ・SEO・セミナー          事例化・推薦・紹介
      │                                     ↑
      ↓                                     │
  Convert（転換）──────────────→ Close（成約）
  問い合わせ・資料DL              提案・受注
  ウェビナー参加
```

**ポイント**: 成約した顧客の成功事例が次の集客コンテンツになり、フライホイールが加速する。

---

### 4.2 チャネルB1: AIチームサービスページ

**目的**: AIチームの存在・サービス内容を伝え、問い合わせを獲得する「ホームベース」

**ページ構成案**:

| セクション | 内容 |
|---|---|
| ヒーロー | キャッチコピー＋AIチームの提供価値を一言で |
| サービス概要 | 4つのパッケージ（研修/PoC/本番実装/伴走）を視覚的に紹介 |
| 導入事例 | 2〜3件の事例をカード形式で（業種・課題・成果） |
| チーム紹介 | メンバーの専門性・実績を紹介（信頼感の醸成） |
| よくあるご質問 | 導入期間・費用感・セキュリティ等のFAQ |
| お問い合わせCTA | フォーム（相談内容を選択式で。フリーテキストも可） |

**問い合わせ対応フロー**:

```
フォーム送信
  → 自動返信メール（受付確認）
  → Slack通知（AIチーム内）
  → 1営業日以内に担当アサイン
  → 3営業日以内に初回コール実施
  → ヒアリング結果に基づきフェーズ判定 → 提案
```

**KPI**:
- ページ訪問数: 月500PV以上（公開6ヶ月後）
- 問い合わせフォーム送信数: 月3件以上
- 問い合わせ→面談設定率: 80%以上

### 4.3 チャネルB2: コンテンツマーケティング

**コンテンツの4本柱**:

| ピラー | 内容例 | ターゲット | 頻度 |
|---|---|---|---|
| **事例コンテンツ** | 「○○業でRAGチャットボット導入、問い合わせ対応時間50%削減」 | 検討中の顧客 | 月1本 |
| **技術コンテンツ** | 「Difyで社内FAQボットを作る方法」「RAG精度改善の5つのコツ」 | 技術担当者 | 月2本 |
| **ビジネスコンテンツ** | 「生成AI導入のROI算出方法」「AI活用ロードマップの作り方」 | 経営層・推進担当 | 月1本 |
| **トレンド解説** | 「2026年の生成AIトレンド」「AI導入補助金の活用法」 | 幅広い層 | 四半期1本 |

**SEOターゲットキーワード（例）**:
- 「生成AI 導入 支援」「RAG 構築 企業」「AI PoC 開発」
- 「業務自動化 AI」「Dify 導入」「n8n ワークフロー」
- 「AI 研修 法人」「ChatGPT 業務活用」

**コンテンツ制作体制**:
- ケーススタディ: 案件完了後にPM＋マーケ担当で作成
- 技術記事: エンジニアメンバーが月1本ローテーション
- ビジネス記事: チームリーダーまたは外部ライターに委託

### 4.4 チャネルB3: セミナー・ウェビナー

**セミナー体系**:

| 種類 | 形式 | 頻度 | 対象 | 目的 |
|---|---|---|---|---|
| **AI活用基礎セミナー** | ウェビナー（60分） | 月1回 | 広く一般 | リード獲得 |
| **業種別AI活用事例セミナー** | ウェビナー（90分） | 四半期1回 | 特定業種 | 質の高いリード獲得 |
| **ハンズオンワークショップ** | 対面 or オンライン（半日） | 四半期1回 | 技術担当者 | 深い関係構築 |
| **NTT東共催セミナー** | 対面（半日） | 半期1回 | NTT東顧客 | NTT東経由のリード獲得 |

**リード獲得フロー**:

```
セミナー告知（Web/メール/NTT東経由）
  → 参加登録（フォーム：会社名・役職・関心分野）
  → セミナー参加
  → アンケート回答（「個別相談を希望しますか？」）
  → 希望者に1週間以内にフォローコール
  → 非希望者にはメールで事後コンテンツ配信
```

**KPI**:
- セミナー参加者数: 各回30名以上
- アンケート回答率: 70%以上
- 個別相談希望率: 20%以上
- セミナー→案件化率: 10%以上

### 4.6 リード管理・ナーチャリング設計

**リードステージの定義**:

| ステージ | 定義 | 判定基準 | 次のアクション |
|---|---|---|---|
| **認知（Awareness）** | サイト訪問・コンテンツ閲覧 | PV、コンテンツDL | メール配信リストに追加 |
| **MQL（Marketing Qualified Lead）** | 関心を示した見込み顧客 | 資料DL、セミナー参加、問い合わせ | 1週間以内にフォローコール |
| **SQL（Sales Qualified Lead）** | 商談可能な見込み顧客 | 初回面談完了、予算・時期が明確 | 提案書作成・見積 |
| **Opportunity** | 提案中の案件 | 提案書提出済み | クロージング活動 |
| **Won / Lost** | 受注 / 失注 | 契約締結 or 辞退 | 受注→事例化 / 失注→原因分析 |

**ナーチャリング施策**:
- MQL向け: 月1回のメールマガジン（事例・セミナー案内）
- SQL未満で滞留: 3ヶ月間のドリップメール（段階的にサービス理解を深める）
- 失注顧客: 半年後にフォローアップ（状況変化の確認）

**管理ツール（最小構成）**:
- Google スプレッドシート（リード管理台帳）
- Slack通知（新規リード・ステージ変更時）
- 将来的にCRM導入を検討（HubSpot Free等）

### 4.7 施策2 構築ロードマップ

| 時期 | アクション | 成果物 |
|---|---|---|
| **Month 1** | サービスページ要件定義・コンテンツ準備 | ページ構成案、原稿ドラフト |

## 5. チャネル間連携（フライホイール）

### 5.1 連携パターン

#### C1. Webリード → NTT東コセル

Webからの問い合わせ顧客がNTT東のテリトリー内にいる場合、NTT東の営業と共同で対応する。

```
Web問い合わせ → 顧客所在地確認 → NTT東担当支店特定
  → NTT東営業と共同訪問 → 成約率UP（NTT東の既存信頼活用）
```

**メリット**: NTT東の契約が絡むことで、顧客側の安心感が増し、AIチーム単独より高い成約率が期待できる。

#### C2. NTT東紹介 → Webコンテンツ活用

NTT東からの紹介案件で、初回面談前に関連するWebコンテンツ（事例・技術記事）を共有し、顧客の理解を事前に高める。

```
NTT東から紹介 → 関連コンテンツをメールで事前共有
  → 初回面談で「記事を見ました」から会話スタート → 商談の質が向上
```

**メリット**: 面談の生産性向上。顧客が事前に知識を得ているため、より具体的な議論ができる。

#### C3. セミナー実績 → 支店向け横展開

ウェビナーの録画やスライドを支店営業の勉強会素材として再利用する。

```
ウェビナー開催 → 録画・スライドを資産化
  → 支店向け勉強会で活用 → 支店営業の理解向上 → 紹介増加
```

**メリット**: コンテンツの一石二鳥。外向けセミナーが内向け啓蒙にもなる。

#### C4. 事例フライホイール

全てのチャネルの成功案件を事例化し、それが全チャネルの燃料になる循環。

```
案件受注（どのチャネル経由でも）
  → 事例コンテンツ化
  → ①Webに掲載（B2）
  → ②紹介資料に追加（A1〜A3向け）
  → ③セミナーで紹介（B3）
  → さらなるリード獲得 → さらなる受注 → さらなる事例
```

**これが最も重要な仕組み。** 事例が増えるほど全チャネルのパフォーマンスが上がる。

## 8. リスク・前提条件

| リスク | 影響 | 対策 |
|---|---|---|
| **AIチームのキャパシティ不足** | インバウンドが増えても対応できない | 優先度の高いチャネル（A1, A2, B1）に集中。低優先は後回し |
| **NTT東との調整に時間がかかる** | 施策1の立ち上がりが遅延 | 並行して施策2（Web）を進め、自力での集客力も確保 |
| **コンテンツ制作のリソース不足** | 施策2が計画通り進まない | 最小限（事例2本＋サービスページ）で開始。外注も検討 |
| **事例公開に顧客許可が得られない** | フライホイールが回らない | 匿名事例でも可。「業種×課題×成果」のパターンで抽象化 |
| **効果測定の仕組みが未整備** | どのチャネルが効いているか分からない | Month 1でリード管理台帳を最低限整備。スプレッドシートで十分 |
