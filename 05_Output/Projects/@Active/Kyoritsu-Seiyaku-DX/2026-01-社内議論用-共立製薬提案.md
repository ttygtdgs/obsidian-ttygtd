### 判断材料

**A: 弊社GCP（SaaS型）を推す場合**
- 先方にクラウド環境がない、またはIT担当者がいない
- 構築がシンプル、短期間で開始可能
- 継続的な収益を確保したい
- Document AIの精度・実績を活かしたい
- データの外部保存に問題がない

**B: 先方AWS + GCP APIを推す場合**
- 先方がデータを自社AWS内に保存したい
- 既存のAWS環境を活用したい
- Document AIの高精度を活かしたい
- GCP新規契約 or 弊社GCP経由でのAPI利用が可能
- ※2クラウドまたぐ構成でやや複雑

**C: 先方AWS完全構築を推す場合** ← 先方がAWS利用中の場合に有力
- 先方がAWSを既に利用中
- 先方がGCP新規契約を避けたい（手続き・管理の手間）
- 既存のAWS環境・契約で全て完結させたい
- コストを抑えたい（Textractは安価: $0.015/ページ）
- 単一クラウドでシンプルな構成を好む
- ※手書き認識精度は要検証（PoCで比較推奨）
- ※弊社のTextractノウハウは限定的（要キャッチアップ）

### 先方がAWSを利用している場合の推奨

先方がAWS環境を利用している場合、以下の順で検討：

1. **手書き認識精度が重要** → パターンB（AWS + GCP API）を推奨
   - Document AIの手書き認識精度を活かす
   - ただしGCP契約が必要

2. **GCP契約を避けたい、コスト重視** → パターンC（AWS完全構築）を検討
   - 先方の既存AWS契約で完結
   - ただしTextractの精度検証が必要

3. **判断がつかない場合** → PoCで両方のOCRエンジンを比較
   - Document AI vs Textractの精度比較を実施
   - 結果に基づいてパターンを決定

```
□ Dropbox自動連携まで含める
□ カスタムレポート追加まで含める
```

> **方針**: 可視化はBIツール（Looker Studio等）を活用し、カスタム管理画面の開発は行わない
> - メリット: 開発コスト削減、短期間で可視化実現、ユーザー自身でカスタマイズ可能
> - デメリット: BIツールの操作習得が必要

## 5. 議論ポイント④: OCR精度検証方法

### 検証データの準備

| 検証フェーズ | データ件数 | 用途 | 準備方法 |
|-------------|-----------|------|---------|
| **学習用** | 30-50件 | Custom Extractorの学習 | 先方提供のサンプルPDF |
| **検証用** | 20-30件 | 精度評価（学習に使用しない） | 学習用と別のサンプル |
| **テスト用** | 10-20件 | 最終確認 | 本番に近い条件のPDF |

### 精度評価の指標

| 指標 | 説明 | 算出方法 |
|------|------|---------|
| **文字認識精度** | 文字単位の認識精度 | 正解文字数 / 全文字数 |
| **項目抽出精度** | 項目（フィールド）単位の精度 | 正解項目数 / 全項目数 |
| **レコード完全一致率** | 全項目が正解のレコード割合 | 完全一致レコード数 / 全レコード数 |
| **人手確認発生率** | 信頼度低で人手確認が必要な割合 | 低信頼度件数 / 全件数 |

### 合格基準（案）

| 指標 | PoC合格基準 | 本開発目標 | 備考 |
|------|------------|-----------|------|
| 項目抽出精度 | 85%以上 | 90%以上 | 主要評価指標 |
| レコード完全一致率 | 70%以上 | 80%以上 | 効率化の実感に直結 |
| 人手確認発生率 | 40%以下 | 20%以下 | 低いほど効率化効果大 |

### 精度検証のプロセス

```
1. 正解データの作成（手動転記で正解を作成）
      ↓
2. OCR処理の実行
      ↓
3. 正解データとの突合（自動比較）
      ↓
4. 精度算出・項目別分析
      ↓
5. エラーパターンの特定
      ↓
6. Custom Extractor再学習（必要に応じて）
      ↓
7. 再検証
```

### 【要議論】精度検証に関する決定事項

```
□ 検証データの準備方法
  - 先方が準備（正解データ込み）
  - 先方が準備（正解データは弊社が作成）
  - 弊社が準備（ダミーデータ作成）

□ 合格基準の設定
  - 上記基準案を採用
  - より厳しい基準を設定
  - 先方と協議して決定

□ 精度が基準に満たない場合の対応
  - PoC終了・見送り
  - 追加学習で改善を試みる（追加費用）
  - 基準を見直して再協議
```

## 6. 議論ポイント⑤: UI合意プロセス

### UI開発の課題

OCRシステムのUI開発では、以下の課題が発生しやすい：

| 課題 | 内容 | 影響 |
|------|------|------|
| **認識のズレ** | クライアントのイメージと成果物が異なる | 手戻り発生 |
| **要件の膨張** | 開発中に追加要望が増える | スケジュール・費用超過 |
| **確認の遅れ** | レビューが遅れ開発が停滞 | 納期遅延 |

### UI合意のタイミング

| フェーズ | 成果物 | 目的 | 確認者 |
|---------|--------|------|--------|
| **要件定義** | ワイヤーフレーム（線画） | 画面構成・機能配置の合意 | 先方担当者 |
| **設計** | モックアップ（カラー付き） | デザイン・操作感の合意 | 先方担当者 |
| **開発中** | プロトタイプ（動くもの） | 実際の操作確認 | 先方担当者 |
| **受入テスト** | 完成版 | 最終確認・承認 | 先方決裁者 |

### ワイヤーフレーム作成の方針

| 選択肢 | 作業量 | 費用 | メリット |
|--------|--------|------|---------|
| **A: 簡易版** | 主要3-5画面のみ | 無償（PoC込み） | 早い |
| **B: 標準版** | 全画面・遷移図付き | XX万円 | 認識ズレ防止 |
| **C: 詳細版** | インタラクション付き | XX万円 | 操作感まで確認可能 |

### レビュー・承認フロー

```
弊社作成 → 社内レビュー → 先方提出 → 先方フィードバック → 修正 → 最終承認
              ↓                           ↓
           1-2日                        3-5営業日
```

### 【要議論】UI合意に関する決定事項

```
□ ワイヤーフレームの作成範囲
  - A: 簡易版（主要画面のみ、PoC込み）
  - B: 標準版（全画面、別途費用）
  - C: 詳細版（インタラクション付き、別途費用）

□ デザインの方針
  - 先方ブランドガイドラインに準拠
  - 弊社標準デザイン
  - フレームワーク標準（Material UI等）

□ レビュー回数の上限
  - 各フェーズ1回まで（追加は別途費用）
  - 各フェーズ2回まで
  - 回数制限なし（工数を見込んで含める）

□ 承認者の明確化
  - 先方担当者レベルで承認
  - 先方決裁者の承認必須
  - 確認が必要（誰が承認権限を持つか）
```

### 想定リスクと対策

| リスク | 発生確率 | 対策 |
|--------|---------|------|
| レビューが長期化 | 中 | 期限を契約に明記 |
| 追加要望が多発 | 高 | スコープを明文化、変更管理プロセス |
| 担当者と決裁者で意見相違 | 中 | 早期に決裁者を巻き込む |
