# 共立製薬 スコアシートDX化 — エンジニア確認用 仕様整理

## 1. プロジェクト概要

### 背景

共立製薬（動物用医薬品メーカー）のワクチン効力試験（チャレンジ試験）において、臨床観察スコアの記録を**紙→iPadでの直接入力**に移行するプロジェクト。

### 現状の業務フロー（AS-IS）

```
巡回 → 紙スコアシート記入 → PDFスキャン → 手作業でExcel転記 → 確認・承認 → Dropbox保存
                                                    ↑
                                              ボトルネック（工数・ミス・属人化）
```

### 目指す姿（TO-BE）

```
巡回 → iPad直接入力（バリデーション付き） → クラウド保存 → Snowflake連携 → Excel出力/ダッシュボード
              ↑                                    ↑
         電子データが原本（ローデータ）       GCP準拠の監査証跡
```

### 規制要件

**厚労省GCP（Good Clinical Practice）準拠が必須**。電子データを原資料（ローデータ）とするため、以下が求められる:

| 要件 | 内容 |
|------|------|
| 監査証跡（Audit Trail） | 全操作（INSERT/UPDATE/DELETE）を記録。変更理由の入力必須 |
| 改ざん防止 | 修正前データの保持、追記のみ（上書き禁止） |
| 電子署名 | 保存時の本人確認（パスワード再入力 or 生体認証） |
| アクセス管理 | ロールベース権限、個人特定可能な認証 |
| バックアップ | 自動バックアップ、災害復旧計画 |

## 4. データモデル

### ER構造

```
study (試験) 1 ──── N score_sheet (個体割当) 1 ──── N observation (観察記録)
```

### 主要テーブル

**study** — 試験マスター
- study_id (PK, UUID), study_name, study_period_start/end, administered_substance, expected_symptoms

**score_sheet** — 試験×個体（紙のスコアシート1枚に対応）
- score_sheet_id (PK, UUID), study_id (FK), animal_id
- UNIQUE(study_id, animal_id)
- administered_substance（個体レベル: 対照群/投与群で異なる）

**observation** — 1回の観察記録
- observation_id (PK, UUID), score_sheet_id (FK), observed_at, observer_name
- 全13項目 + 体温・体重 + 備考
- UNIQUE(score_sheet_id, observed_at)
- created_by（監査証跡用）

### DB基盤

- **分析基盤**: Snowflake（共立製薬が既に利用中）
- **トランザクションDB**: Cloud SQL (PostgreSQL) を想定

## 5. アーキテクチャ構成（案）

```
┌─────────────────────┐          ┌──────────────────────────┐
│   iPad アプリ        │          │        クラウド            │
│                     │          │                          │
│  入力フォーム       │  sync    │  API Server (Cloud Run)  │
│  写真撮影          │ ←─────→ │         │                │
│  オフラインDB      │          │  Cloud SQL (PostgreSQL)  │
│  (SQLite/Realm)    │          │         │                │
│  監査証跡ログ(local)│          │  Cloud Storage (写真)    │
│                     │          │         │                │
└─────────────────────┘          │  Snowflake (分析基盤)    │
                                 └──────────────────────────┘
```

### 技術選定（案）

| レイヤー | PoC | 本開発 | 理由 |
|---------|-----|--------|------|
| iPad アプリ | **PWA (Web)** | Swift or Flutter | PoC=素早く検証、本開発=UX・オフライン安定性 |
| バックエンド | Cloud Run (Python/FastAPI) | 同左 | GCP/AWS環境、スケーラブル |
| DB | Cloud SQL (PostgreSQL) | 同左 | 監査証跡・トランザクション対応 |
| 写真保存 | Cloud Storage | 同左 | 大容量・低コスト |
| 分析基盤 | Snowflake | 同左 | 先方既存利用中 |

## 6. 主要機能要件

### 6.1 入力バリデーション

| チェック | 内容 | タイミング |
|---------|------|-----------|
| 範囲チェック | 体温35.0〜42.0℃等 | 入力即時 |
| スコア値域 | 各項目の最大値超えチェック | 入力即時 |
| 異常値警告 | 体温≧39.5℃等で⚠表示 | 入力即時 |
| 急変検知 | 前日比で大きな変化がある場合に注意喚起 | 保存時 |
| 必須項目 | 観察者名、観察日時 | 保存時 |
| 未入力チェック | 全項目空の場合に確認ダイアログ | 保存時 |

### 6.2 オフライン対応

| 項目 | 仕様 |
|------|------|
| 入力 | オフラインで全操作可能 |
| ローカル保存 | SQLite/Realm等でデータ・写真を保存 |
| 同期 | ネットワーク復帰時に自動同期 |
| 競合解決 | タイムスタンプベースで新しいデータを優先（競合時は通知） |
| 同期状態表示 | ✓同期済 / ⏳未同期 n件 |

### 6.3 監査証跡（Audit Trail）— **GCP査察対応の最重要機能**

| 記録内容 | 詳細 |
|---------|------|
| 操作種別 | INSERT / UPDATE / DELETE |
| 操作者 | ログインユーザー名 |
| 操作日時 | 端末ローカル時刻 + UTC |
| 変更項目 | フィールド名 |
| 変更前/後の値 | 修正前・修正後のデータ |
| **変更理由** | **修正時は理由入力を必須** |
| 電子署名 | 保存時にパスワード再入力で本人確認 |

### 6.4 ワークフロー（承認フロー）

```
観察者: スコア入力 → 保存（電子署名）
         ↓
試験責任者: データ確認 → 承認（電子署名）
         ↓
ロック状態（以降は修正理由付きの訂正のみ可能）
```

### 6.5 写真撮影

| 機能 | 仕様 |
|------|------|
| 撮影 | iPadカメラ、1観察あたり複数枚可 |
| 異常時の自動促し | 臨床スコアが閾値以上で写真撮影を促すダイアログ |
| メタデータ自動付与 | 個体番号、Day、日時、撮影者を自動紐付け |
| 保存 | ローカル保存 → Wi-Fi復帰時にクラウドアップロード |

## 8. 懸念事項・技術的リスク

### 8.1 GCP準拠の監査証跡

**リスク: 高**

- 厚労省GCP準拠のAudit Trailは、医薬品治験システム（EDC）の領域であり、要件が厳格
- 「全操作の記録」「修正理由の必須化」「電子署名」「改ざん防止」をすべて満たす必要
- **21 CFR Part 11（FDA）に近い要件**が求められる可能性あり（動物薬の場合の適用範囲要確認）
- 監査証跡の実装が不十分だと、査察時に**システム全体が否認される**リスク

**確認すべき点:**
- 動物用医薬品の場合、厚労省GCPの適用範囲はどこまでか（人用医薬品と同等か軽微か）
- 既存のEDCシステム（REDCap等）の利用は検討済みか → カスタム開発 vs 既存パッケージ
- バリデーション（CSV: Computerized System Validation）の実施が必要か

### 8.2 オフライン同期の競合解決

**リスク: 中**

- 複数iPadからの同時入力で競合が発生する可能性
- 特に**同一個体を複数の観察者が入力**するケースがあるか要確認
- タイムスタンプベースの競合解決だけでは不十分な可能性
- **監査証跡が絡むため、同期時のデータ整合性は非常にクリティカル**

**確認すべき点:**
- 同一個体を複数人が同時に入力するケースはあるか
- 同期失敗時のリカバリ手順（ローカルデータの保護）

### 8.3 PWA vs ネイティブのギャップ（PoC→本開発移行）

**リスク: 中**

- PoCをPWA、本開発をネイティブ（Swift/Flutter）で想定しているが:
  - PWAのオフライン対応はService Worker依存で制約が多い（iOS Safari特有の制限）
  - PWAのカメラ制御は限定的
  - PoCで得た知見がネイティブ移行時にどこまで活きるか
- PWAで十分なら本開発もPWAのままという判断もありうるが、GCP準拠の厳格さとの兼ね合い

**確認すべき点:**
- iOSのPWAでService Workerの永続化（バックグラウンド同期）がどこまで可能か
- カメラ撮影のメタデータ取得がPWAで十分か

### 8.4 ±スコアのデータ表現

**リスク: 低〜中（ただし全体に波及）**

- 臨床スコアの「±」をDB上でどう扱うかが未決定
- 選択肢:
  - A: 整数マッピング（±=0 or ±=-1）→ 集計は容易だが元表記が失われる
  - B: VARCHAR型で保持 → 集計時にCAST必要
  - C: 2カラム方式（raw + numeric）→ カラム数増大
- **全テーブル・API・UIに影響するため早期決定が必要**

### 8.5 既存Snowflake環境との連携

**リスク: 中**

- 共立製薬が既にAWS + Snowflakeを利用中だが:
  - 弊社へのアクセス権限付与の範囲が未定
  - 情シス部門との連携がまだ始まっていない
  - Snowflake連携のタイミング（リアルタイム or バッチ）未決定
- 情シスの協力が得られないと、データ基盤連携部分がブロッカーになる

### 8.6 セキュリティ・認証

**リスク: 中**

- 実験動物の臨床データは機密性が高い
- ユーザー管理方式が未定（固定アカウント / AD連携 / 既存認証基盤）
- 共有iPadの場合、セッション管理・自動ログアウトの設計が複雑化
- 手袋着用環境でのFace ID/Touch IDの動作確認が必要

### 8.7 写真データの容量とストレージコスト

**リスク: 低**

- 撮影タイミングが「毎日全個体」の場合、ストレージが大きくなる
- 1枚あたり2-5MB × 個体数 × 日数 × 試験期間
- Cloud Storageのライフサイクル管理（ホット→コールド移行）の設計が必要

## 9. 開発スケジュール（案）

```
Phase 0: 仕様確定                [2週間]
├─ クライアント確認事項のすり合わせ
├─ UI/UXプロトタイプ（Figma）
└─ 技術検証

Phase 1: PoC開発                 [4〜6週間]
├─ PWAベースの入力フォーム
├─ 主要項目の入力・保存
├─ オフライン基本動作
├─ 写真撮影（基本）
└─ 現場での試用・フィードバック収集

Phase 2: 本開発                  [3〜4ヶ月]
├─ ネイティブ or Flutter（PoCの結果で判断）
├─ 全機能実装
├─ 監査証跡・承認フロー
├─ Snowflake連携
├─ Excel出力
└─ セキュリティ対応

Phase 3: 導入・運用              [1ヶ月]
├─ ユーザートレーニング
├─ 並行運用（紙 + iPad）
├─ 運用マニュアル
└─ 本番稼働
```

## 11. エンジニアへの確認依頼事項

以下の観点でレビュー・意見をいただきたい:

### アーキテクチャ

- [ ] PWA → ネイティブの2段階アプローチは妥当か？最初からFlutterで行くべきか？
- [ ] オフライン同期のアーキテクチャ（SQLite + API同期）で問題ないか？
- [ ] Cloud SQL (PostgreSQL) + Snowflake の二重構成は妥当か？Snowflake直接でいけるか？

### GCP準拠

- [ ] 監査証跡の実装方針（DB層のトリガー vs アプリ層のイベントログ）
- [ ] 既存のEDCパッケージ（REDCap等）の利用可否の検討
- [ ] CSV（Computerized System Validation）の必要性と対応コスト

### データモデル

- [ ] ±スコアの扱い方の推奨（整数マッピング vs VARCHAR vs 2カラム方式）
- [ ] 監査証跡テーブルの設計（別テーブル？CDC？Event Sourcing？）
- [ ] 写真メタデータの管理方式

### 見積もり

- [ ] PoC 200〜400万円、本開発 800〜1,500万円のレンジ感は妥当か？
- [ ] GCP準拠の監査証跡対応で見積もりが膨らむリスク
