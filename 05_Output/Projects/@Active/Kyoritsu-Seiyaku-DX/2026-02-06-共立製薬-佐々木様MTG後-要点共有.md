# 【共有】共立製薬 佐々木様ヒアリング結果

## 概要

共立製薬・佐々木様との会話により、電子化提案に関する先方の意向が明確になった。
以下に要点を整理する。

---

## 先方の意向（確認済み）

| # | 論点 | 先方スタンス |
|---|------|-------------|
| 1 | **稟議の進め方** | 佐々木様ら現場で解像度を上げた上で、経営層へ稟議を出す方向で**合意** |
| 2 | **紙へのこだわり** | **なし** → デジタル入力への移行に前向き |
| 3 | **ローデータの保管** | **こだわりあり** → 紙・データ問わず、ローデータの保管が必須 |

### ローデータ保管の背景

- NTT東日本側で厚労省サイトを確認済み → 紙でもデータでも、ローデータの保管が必要
- 参照: https://www.mhlw.go.jp/web/t_doc?dataId=00tc5323&dataType=1&pageNo=
- 当該ページは厚労省GCP（Good Clinical Practice / 医薬品臨床試験の基準）に基づく治験文書保管の規定
- **使用する媒体を問わず**、原資料（ローデータ）の保管義務がある

## Part 1: NTTDXPNとしての提案方針（社内整理）

### 提案の全体構成（2本柱）

| 施策 | 概要 |
|------|------|
| **提案①** スコアシートDX化 | スマートデバイス入力への移行（PoC → 本開発） |
| **提案②** データ基盤構築 | Snowflake活用によるデータ利活用基盤の構築 |

### 提案①：スコアシートDX化の方針

#### 把握済みの情報

| 項目 | 内容 | 提案への影響 |
|------|------|-------------|
| スコアシート種類 | **1種類** | フォーム設計のスコープが明確。1フォームで完結 |
| 項目数 | **10-20項目** | 比較的シンプル。スマートデバイスの1画面に収まる可能性あり |
| 記入内容 | **数値（スコア）が中心** | 数値入力フォーム + バリデーションで品質担保しやすい |
| 通信環境 | **ネットワークあり** | オフライン対応は優先度低 → 開発コスト抑制できる |
| 紙のこだわり | **なし** | スマートデバイスでの直接入力が有力 |
| ローデータ | **こだわりあり** | 電子データを原本とする設計。監査証跡が必須 |

#### 弊社の方針

- **入力方式**: スマートデバイス（タブレット or スマートフォン）での直接入力
- **技術方針**: 1種類・10-20項目・数値中心 → シンプルな入力フォームで実現可能。大規模開発にはならない
- **ローデータ対応**: 電子データを原本として設計。厚労省GCP（Good Clinical Practice）に基づく監査証跡・改ざん防止を組み込む
- **進め方**: PoC → 本開発の2段階。PoCで入力フォームの試作と現場フィードバックを得る

#### アプリ構築の論点

| # | 論点 | 選択肢 | 判断材料 |
|---|------|--------|---------|
| 1 | **プラットフォーム** | Webアプリ（PWA） / ネイティブアプリ / ハイブリッド | ネットワークあり→PWAで十分な可能性。配布・更新のしやすさも考慮 |
| 2 | **対象デバイス** | タブレット / スマートフォン | 10-20項目を1画面に収めるならタブレットが有利。現場の持ち運びやすさとのトレードオフ |
| 3 | **デバイス調達** | 先方既存デバイス / 新規調達 | コストに大きく影響。**先方に確認が必要** |
| 4 | **入力フォーム設計** | 紙のレイアウト踏襲 / デバイス最適化UI | 現場の慣れを重視するか、操作性を重視するか。PoCで検証すべき |
| 5 | **バリデーション** | リアルタイム検証 / 送信時検証 | 数値中心→入力時に範囲チェック・必須チェックを入れることでデータ品質を担保 |
| 6 | **確認・承認フロー** | 入力者のみ / 入力者→承認者の2段階 | 現状の紙でも確認工程がある→**システム化後も確認フローは維持**。承認者の権限設計が必要 |
| 7 | **ユーザー認証** | ID/パスワード / SSO / 生体認証 | 監査証跡で「誰が入力したか」を記録するため、個人を特定できる認証が必須 |
| 8 | **監査証跡（Audit Trail）** | 全操作ログ / 修正のみ記録 | GCP要件：データ修正履歴の記録が必須。入力・修正・削除すべてを記録する設計 |
| 9 | **データ修正ポリシー** | 上書き禁止（修正は新レコード） / 上書き可（履歴付き） | GCP要件：修正前のデータも残す必要あり。「誰が・いつ・何を・どう変えたか」を記録 |
| 10 | **データ保存先** | AWS（先方既存） / 別クラウド | 提案②のSnowflakeとの連携を考えるとAWSが自然 |
| 11 | **Excel出力** | アプリから直接出力 / データ基盤経由 | 既存業務との互換性。現行フォーマットに合わせた出力が必要か確認 |
| 12 | **PoCのスコープ** | 入力フォームのみ / 入力+確認フロー / 入力+確認+監査証跡 | PoCでどこまで作るかで費用・期間が変わる。**NTT東と合意が必要** |

#### 論点の優先度

**PoC前に決めるべき（NTT東との擦り合わせ対象）**
- #1 プラットフォーム → 開発言語・技術スタックが決まる
- #2 対象デバイス → フォーム設計に直結
- #6 確認・承認フロー → 業務設計の根幹
- #12 PoCスコープ → 費用・期間の前提

**PoC中に検証すべき**
- #4 入力フォーム設計 → 現場フィードバックで決める
- #5 バリデーション → 実データで検証

**本開発フェーズで詳細設計**
- #7 ユーザー認証 → 情シスとの連携が必要
- #8 #9 監査証跡・修正ポリシー → GCP要件の詳細確認後に設計
- #10 データ保存先 → 情シスとのインフラ設計
- #11 Excel出力 → 既存フォーマットの確認後

### 提案②：データ基盤構築の方針

#### 把握済みの情報

| 項目 | 内容 |
|------|------|
| 先方インフラ | AWS + Snowflake（利用中） |
| 情シス連携 | NTT東側で接点あり |

#### 弊社の方針

- **基盤**: 先方既存のAWS + Snowflakeを活用
- **情シス連携**: NTT東経由で接点を活かし、顔合わせから進める
- **将来連携**: 提案①のデバイス入力データ → Snowflakeに直接投入する構成を見据える

### 費用方針

| 項目 | 方針 |
|------|------|
| **費用負担** | 共立製薬負担。NTTDXPNは開発受託として費用をいただく |
| **概算費用** | これから検討（スマートデバイス方式での再見積が必要） |
| **PoC費用** | 共立製薬負担。稟議のハードルを考慮した金額設定が必要 |

## Part 2: NTT東との擦り合わせ議題

### NTT東の確認ポイント（6項目）に対する準備状況

| # | NTT東の確認ポイント | 弊社準備状況 |
|---|---------------------|-------------|
| ① | 今の業務フロー（AS-IS） | 把握済み：巡回→紙記入→PDFスキャン→Excel手転記→**確認・承認**→Dropbox保存 |
| ② | システム化後のフロー（TO-BE） | 把握済み：巡回→スマートデバイス直接入力→**確認・承認**→クラウド保存→Excel出力/ダッシュボード（※確認フローは残す。転記作業がなくなるだけで、データの正確性を担保する確認工程は維持） |
| ③ | 蓄積できるデータ | 回答準備可：入力データ（スコア）、入力メタデータ（日時・場所・担当者）、監査証跡 |
| ④ | データの活用方法 | 回答準備可：ダッシュボードでの可視化、傾向分析、Snowflakeでの検索・集計 |
| ⑤ | 受益者 | 回答準備可：現場担当者（転記不要）、管理者（リアルタイム把握）、経営層（データ活用） |
| ⑥ | 決裁者メリット・ROI | **要検討**：ROI算出には先方の定量データが必要（下記参照） |

### 先方に確認が必要な事項（ROI算出・提案具体化に必須）

#### ROI算出のための定量情報

| 確認事項 | 理由 | 確認先 |
|---------|------|--------|
| 月間の処理件数 | 削減効果の定量化に必要 | 佐々木様 |
| 1件あたりの転記作業時間 | 工数削減額の算出根拠 | 佐々木様 |
| 転記担当者は何名か | 属人化リスクの定量化、人件費削減の試算 | 佐々木様 |

→ **⑥のROIを出すには、この3点が最低限必要。NTT東経由で佐々木様に確認したい。**

#### ローデータ要件の深掘り

| 確認事項 | 理由 | 確認先 |
|---------|------|--------|
| 電子データを原本とすることへの組織的合意範囲 | 佐々木様個人の了解だけでなく、品質管理部門等の承認も必要か | 佐々木様 |
| 現在のローデータ保管方法と保管期間 | 電子化後のシステム要件に直結 | 佐々木様 |
| 監査証跡・改ざん防止の要件レベル | どこまで厳密に対応するかでコストが変わる | 佐々木様 + 情シス |

#### 現場のフィージビリティ

| 確認事項 | 理由 | 確認先 |
|---------|------|--------|
| 現場スタッフのデバイス習熟度 | UI設計の難易度、教育コストに影響 | 佐々木様 |
| 既存デバイスの有無 | 新規調達 or 既存利用でコストが大きく変わる | 佐々木様 |

#### 移行計画（経営層向けリスク回答の準備）

| 確認事項 | 理由 | 確認先 |
|---------|------|--------|
| 紙→電子への切替期間（並行運用の有無） | 移行リスクの評価 | 佐々木様 + 弊社で設計 |
| 既存の紙ローデータの取り扱い | 遡ってデジタル化するか、ある時点から切り替えるか | 佐々木様 |
| システム障害時のフォールバック | 決裁者は「失敗したらどうする？」を必ず聞く | 弊社で設計 |

### NTT東に依頼・相談したい事項

| # | 依頼・相談事項 | 背景 |
|---|---------------|------|
| 1 | **佐々木様への定量データ確認（月間件数・工数・人数）** | ROI算出に必須。次回MTGまでに確認いただきたい |
| 2 | **情シス連携の進め方** | NTT東に接点あり。顔合わせの段取りを相談したい |
| 3 | **費用見積の分担** | 概算費用のレンジをどう組み立てるか。NTT東側のコスト構造との整合 |
| 4 | **稟議資料の作成分担** | 事業部長レベルの稟議。NTT東・NTTDXPNでどう分担するか |
| 5 | **スケジュール感の合意** | PoC→本開発のスケジュールをNTT東と合意した上で先方に提示したい |
